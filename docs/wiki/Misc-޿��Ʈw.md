## ESG 數據總覽 [ESG Metrics Dashboard]
路徑： /metrics/dashboard | 權限： PM, CSO, Analyst | 所屬旅程： I. 數據採集與監控

1.  模組定位 (Core Purpose)
    ESG 數據總覽提供企業核心 ESG 指標的綜合性儀表板，透過視覺化圖表與即時數據，幫助使用者快速掌握永續績效、追蹤趨勢並支援數據驅動的決策。

2.  客戶旅程與 UX 體驗 (Customer Journey & UX)
    目標對象與痛點： 永續專案經理 (PM) 或分析師。他們需要快速掌握企業在各 ESG 面向的表現，但傳統報表分散、更新不及時，難以進行趨勢分析或跨指標比較，導致決策延遲或不準確。
    體驗高光時刻 (Aha Moment)： 當 PM 在儀表板上篩選「2023 年 Q3」並點擊「碳排放」指標時，看到範疇一、二、三的排放量趨勢圖、碳密集度與同業比較數據在數秒內動態呈現，並自動標示出異常波動，感受到「所有關鍵數據一目瞭然，決策信心大增」。
    操作軌跡：
    1.  進入 ESG 數據總覽頁面。
    2.  選擇時間範圍（例如：年度、季度、自定義區間）。
    3.  點擊「GHG Emissions」卡片，查看詳細排放數據與趨勢圖。
    4.  切換至「Energy Consumption」頁籤，比較再生能源比例與總用電量。
    5.  利用篩選器，針對特定廠區或事業部進行數據鑽取，並將結果匯出為 PDF 報告。

3.  UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
    桌面版佈局： 採用「多卡片式儀表板 (Card-based Dashboard)」佈局，頂部為全局篩選器與時間軸控制，下方為可拖曳、自定義大小的數據卡片 (如：趨勢圖、圓餅圖、指標數字、表格)。
    核心液態玻璃元件： BrandMetricCard (顯示關鍵指標與趨勢箭頭)、BrandLineChart (多系列折線圖)、BrandPieChart (圓餅圖)、BrandFilterBar (全局篩選器)、BrandDataTable (可排序、篩選的數據表格)。
    行動端適配 (RWD)： < 768px 時，卡片自動堆疊成單欄佈局，篩選器收合為底部 Tab 或懸浮 FAB。圖表縮放以保持可讀性，並提供點擊放大查看詳情的功能。複雜表格自動轉化為堆疊式卡片或提供橫向滾動。

4.  核心邏輯與 5T 協定 (Logic & 5T Protocol)
    資料流向： 從 `ghg_emissions`, `energy_consumption`, `water_waste`, `social_metrics`, `governance_metrics` 等資料表撈取原始數據，經後端聚合計算後，透過 GraphQL API 傳輸至前端渲染。任何數據更新會即時影響儀表板 KPI。數據匯出功能會觸發後端報告生成服務。
    5T 實踐點：
    [T1 Tangible 具體]： 將抽象的 ESG 數據（如排放量、用水量、女性比例）具體化為可視化的圖表、百分比和趨勢線，讓使用者直觀理解其意義和變化，並能與設定的目標值進行比較。
    [T3 Trackable 追蹤]： 儀表板上的所有數據點都可追溯至其原始輸入來源（例如：點擊碳排放數據可連結至「證據金庫」中的電費單或燃料採購紀錄），並顯示數據的最後更新時間與操作者。
    [T4 Transparent 透明]： 數據計算邏輯（如碳密集度計算公式、範疇邊界定義、FR/SR 指標計算方式）在頁面中提供資訊提示 (Tooltip) 或連結至說明文件，確保計算過程公開透明且可供審閱。

5.  功能項目解說和使用技術 (Features & Tech Stack)
    數據篩選與時間軸控制： 允許使用者按時間、部門、廠區、指標類型等維度篩選數據。技術使用 React Hook Form 處理表單狀態，搭配 React Date Range Picker 和多選下拉選單。
    互動式數據圖表： 支援縮放、Tooltip、鑽取、多系列比較等功能。技術使用 Recharts 或 ECharts 進行圖表渲染，結合 React Query 進行數據緩存與同步，確保數據加載流暢。
    KPI 異常警示與目標追蹤： 根據預設閾值、歷史數據或設定的目標值，自動標示出異常波動或未達標的指標。技術使用後端 Python 服務 (如 FastAPI) 執行數據分析與異常檢測演算法，前端透過 WebSocket 或 SSE 接收警示並即時更新 UI。
    報告匯出與排程： 支援將當前儀表板視圖匯出為 PDF/Excel 報告，並可設定定期自動生成報告並發送至指定郵箱。技術使用 Puppeteer 或 Headless Chrome 進行前端渲染截圖，結合後端報告生成服務 (如 LibreOffice)。

6.  品質達標與驗收紅線 (QA Red Lines)
    🚨 UI 跑版紅線： 在任何解析度下，數據卡片內容（數字、圖表、文字）必須完整顯示，不可出現溢出或重疊，特別是圖表軸標籤與圖例。卡片拖曳與調整大小後，佈局必須保持穩定，不可出現元件錯位。
    🚨 邏輯/體驗紅線： 當篩選條件導致無數據時，頁面必須顯示友善的「無數據」提示與建議操作，而非空白或錯誤訊息。數據加載時間在 3 秒內完成，否則必須顯示骨架屏 (Skeleton Screen) 或加載動畫，避免白屏。所有圖表數據必須與原始數據表保持一致，不可出現計算錯誤或數據不匹配。

## 治理與稽核中心 [Governance & Audit Hub]
路徑： /audit/hub | 權限： CSO, PM, Auditor | 所屬旅程： V. 確信審計與發佈

1.  模組定位 (Core Purpose)
    治理與稽核中心是企業永續報告的誠信保障，提供佐證文件管理、操作日誌追溯與重大性議題評估功能，確保所有 ESG 數據與決策的透明度、可追溯性與可信度。

2.  客戶旅程與 UX 體驗 (Customer Journey & UX)
    目標對象與痛點： 永續長 (CSO)、永續專案經理 (PM) 及外部稽核員。CSO 需要確保所有數據符合法規且經得起審查；PM 則需高效管理大量佐證文件；稽核員則希望快速驗證數據來源與變更歷史，避免耗時的人工比對。
    體驗高光時刻 (Aha Moment)： 當稽核員在「操作日誌」中點擊某筆數據變更紀錄，系統不僅顯示變更前後的值，還自動連結到「證據金庫」中對應的原始憑證，並顯示該憑證的 ZKP 驗證狀態。稽核員瞬間感受到「所有疑慮都被預先解答」的安心感。
    操作軌跡：
    1.  稽核員進入「治理與稽核中心」。
    2.  點擊「重大性議題」頁籤，檢閱本年度的雙重重大性矩陣與評估過程。
    3.  切換至「操作日誌」，篩選特定時間段或操作者，查看數據變更記錄。
    4.  點擊某筆數據變更紀錄，查看詳細的變更內容與相關佐證。
    5.  在「證據金庫」中，上傳新的佐證文件，並將其關聯至相關的 ESG 數據點。

3.  UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
    桌面版佈局： 採用「多頁籤式佈局 (Tabbed Layout)」，包含「證據金庫」、「操作日誌」與「重大性議題」三個主要頁籤。每個頁籤內部採用表格或互動式圖表佈局。
    核心液態玻璃元件： VaultOmniTable (證據金庫表格，包含預覽與狀態標籤)、AuditLogTable (操作日誌表格，可展開詳情)、MaterialityMatrix (互動式重大性矩陣圖)、BrandStatusDot (顯示 ZKP 驗證狀態)。
    行動端適配 (RWD)： < 768px 時，頁籤轉化為底部 Tab 或下拉選單。表格自動轉化為「堆疊式資料卡片 (Stacked Cards)」，隱藏複雜的 Hash 字串，僅保留關鍵資訊。重大性矩陣提供「點擊展開圖表」的橫向全螢幕選項。

4.  核心邏輯與 5T 協定 (Logic & 5T Protocol)
    資料流向：
    *   `evidence_vault`：儲存佐證文件的元數據（檔名、Hash、ZKP 狀態、大小、GRI 條款），實體檔案上傳至 Supabase Storage / S3。
    *   `audit_logs`：記錄所有使用者對 ESG 數據和系統配置的操作行為（操作者、時間、行為、變更內容、相關數據 ID）。
    *   `materiality_topics`：儲存 ESG 議題名稱、財務衝擊 (Impact) 與環境衝擊 (Concern) 分數，以及雙重重大性定位。
    所有資料變更都會觸發 `audit_logs` 寫入。
    5T 實踐點：
    [T2 Traceable 追溯]： 每一筆 ESG 數據的變更，都會在 `audit_logs` 中留下不可篡改的紀錄，包含操作者、時間、變更前後的值，並可追溯至相關的佐證文件 ID。
    [T4 Transparent 透明]： 重大性議題的評分來源與計算邏輯（如 Impact/Concern 的權重）在介面中清晰呈現，並允許查看歷史版本的矩陣，確保評估過程公開。
    [T5 Trustworthy 信任]： 「證據金庫」中的文件在客戶端上傳時即計算 SHA-256 Hash，並隨檔案儲存。任何對檔案的修改都會導致 Hash 不匹配，系統會立即標示為「已損毀/失效」，確保文件內容的完整性與不可篡改性。

5.  功能項目解說和使用技術 (Features & Tech Stack)
    證據金庫管理： 支援多檔案拖曳上傳、檔案預覽、關聯 ESG 數據點、ZKP 狀態顯示。技術使用 react-dropzone 進行拖曳上傳，PDF.js 進行 PDF 預覽，Web Crypto API 進行客戶端 Hash 運算，Supabase Storage 進行檔案儲存。
    操作日誌查詢與過濾： 提供多維度篩選（時間、操作者、操作類型、受影響數據），並可展開查看詳細變更內容。技術使用 React Table 搭配後端 Supabase PostgreSQL 的全文檢索與索引優化。
    互動式重大性矩陣： 允許 CSO 調整 Impact/Concern 分數滑桿，即時觀察氣泡圖的動態變化，並支援「鎖定矩陣」功能。技術使用 Recharts 或 D3.js 進行圖表渲染，結合 Zustand 或 Context API 進行狀態管理，Supabase RLS 實現鎖定後的權限控制。

6.  品質達標與驗收紅線 (QA Red Lines)
    🚨 UI 跑版紅線： 「證據金庫」表格中，檔名或 Hash 字串過長時必須觸發 text-overflow: ellipsis (顯示...)，配合 Tooltip 檢視完整內容，絕對不可撐開表格導致整個頁面出現醜陋的 X 軸滾動條。重大性矩陣中，氣泡重疊時 Tooltip 必須具備「智慧偏移 (Smart Placement)」邏輯，不可互相遮擋。
    🚨 邏輯/體驗紅線： 「鎖定矩陣」點擊後，必須彈出二次確認對話框。鎖定後任何嘗試拉動滑桿的行為都應給予「已鎖定」的視覺阻尼回饋 (Visual Spring Feedback)，而非按鈕突然消失導致突兀感。若系統檢測到使用者上傳了重複的檔案 (Hash 完全相同)，必須阻斷上傳並彈出友善提示：「此文件已存在於金庫中，是否直接建立關聯？」，以節省儲存空間與防止資料冗餘。

## 任務與營運管理 [Tasks & Operations Management]
路徑： /operations/tasks | 權限： ALL_USERS (for their own tasks), PM, Admin | 所屬旅程： III. 任務分派與協作

1.  模組定位 (Core Purpose)
    任務與營運管理模組提供一個集中化的平台，用於分派、追蹤和管理所有與 ESG 相關的任務，確保各部門協同合作，高效推進永續目標的達成。

2.  客戶旅程與 UX 體驗 (Customer Journey & UX)
    目標對象與痛點： 各部門填報窗口、永續專案經理 (PM)。填報者常常忘記截止日期或不清楚任務優先級；PM 則難以全面掌握各項任務的進度與瓶頸，導致專案延誤。
    體驗高光時刻 (Aha Moment)： 當 PM 在任務看板上將「收集電費單」任務從「進行中」拖曳到「已完成」時，系統自動發送通知給相關負責人，並更新整體專案進度條。PM 看到任務狀態即時更新，感受到「專案進度盡在掌握」的掌控感。
    操作軌跡：
    1.  PM 進入「任務與營運管理」頁面，查看所有待辦任務。
    2.  篩選出「逾期」任務，並點擊其中一項，查看詳細資訊與歷史溝通紀錄。
    3.  將一個新任務分派給特定負責人，設定截止日期與優先級。
    4.  填報窗口收到任務通知，點擊進入任務詳情頁面，上傳相關佐證文件。
    5.  完成任務後，將任務狀態更新為「已完成」。

3.  UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
    桌面版佈局： 採用「看板式 (Kanban Board)」與「列表式 (List View)」切換佈局。看板式以任務卡片呈現，按狀態（待辦、進行中、已完成）分欄；列表式則為可排序、篩選的表格。
    核心液態玻璃元件： BrandTaskCard (顯示任務標題、負責人、截止日期、進度條)、BrandKanbanColumn (看板欄位)、BrandProgressRing (圓形進度環)、BrandNotificationBadge (通知角標)。
    行動端適配 (RWD)： < 768px 時，看板式自動轉為「單欄垂直滾動」，任務卡片內容精簡。列表式表格轉化為「堆疊式資料卡片」，僅顯示核心資訊，點擊展開詳情。

4.  核心邏輯與 5T 協定 (Logic & 5T Protocol)
    資料流向： 任務的創建、更新、狀態變更等操作，都會即時寫入 `tasks` 資料表。任務的進度與狀態會影響相關的 ESG 數據採集進度，並可能觸發 `notifications` 資料表的寫入。
    5T 實踐點：
    [T1 Tangible 具體]： 將抽象的「永續專案」拆解為具體的、可執行的任務，並透過進度條、狀態標籤和截止日期，讓使用者清晰看到任務的實體進展。
    [T2 Traceable 追溯]： 每個任務都包含完整的操作歷史紀錄（誰在何時做了什麼），並可連結到相關的 ESG 數據點或「證據金庫」中的佐證文件，確保任務執行的可追溯性。
    [T4 Transparent 透明]： 任務的狀態、負責人、截止日期和進度對所有相關成員公開透明，PM 可以一目瞭然地掌握整個專案的健康狀況。

5.  功能項目解說和使用技術 (Features & Tech Stack)
    任務創建與分派： 支援任務標題、描述、負責人、截止日期、優先級、關聯 ESG 指標等設定。技術使用 React Hook Form 進行表單管理，搭配 React Date Picker 和多選用戶選擇器。
    看板與列表視圖切換： 提供靈活的任務展示方式，支援拖曳排序與狀態變更。技術使用 React Beautiful DND 實現看板拖曳功能，React Table 實現列表視圖。
    任務進度追蹤與提醒： 根據任務狀態自動更新進度，並在截止日期臨近或逾期時發送系統通知/郵件提醒。技術使用 Supabase Realtime 監聽 `tasks` 表變更，後端 Cron Job 服務定期檢查任務狀態並觸發通知服務。
    任務評論與附件： 允許使用者在任務詳情頁面進行討論，並上傳相關附件。技術使用 Supabase Storage 儲存附件，Supabase Realtime 實現即時評論功能。

6.  品質達標與驗收紅線 (QA Red Lines)
    🚨 UI 跑版紅線： 看板模式下，當單一任務欄位內任務卡片過多時，必須正常出現 Y 軸滾動條，不可撐破欄位容器。任務卡片內容（標題、負責人、日期）過長時必須觸發 text-overflow: ellipsis。
    🚨 邏輯/體驗紅線： 任務狀態變更後，相關的專案進度條必須在 1 秒內更新。當分派任務給不存在的用戶時，必須給予明確的錯誤提示。任務提醒必須在設定時間點準時發送，不可延遲或遺漏。

## 利害關係人與供應鏈管理 [Stakeholder & Supply Chain Management]
路徑： /external/relations | 權限： PM, Procurement, CSO | 所屬旅程： II. 策略盤點與分派

1.  模組定位 (Core Purpose)
    利害關係人與供應鏈管理模組提供全面的工具，用於識別、分析、互動和管理企業的關鍵利害關係人與供應商，確保永續發展策略能有效回應外部期望並降低供應鏈風險。

2.  客戶旅程與 UX 體驗 (Customer Journey & UX)
    目標對象與痛點： 永續專案經理 (PM)、採購部門主管、永續長 (CSO)。PM 需要系統化管理複雜的利害關係人網絡；採購主管則需評估供應商的 ESG 表現；CSO 則希望快速識別高風險供應商或關鍵利害關係人的關注議題，但傳統方式耗時且數據分散。
    體驗高光時刻 (Aha Moment)： 當採購主管在供應商列表上篩選「高風險」供應商時，系統不僅即時顯示這些供應商的 ESG 評分、風險等級，還自動標示出其未簽署的「永續承諾書」。主管瞬間感受到「供應鏈風險一目瞭然，決策有據可依」的效率。
    操作軌跡：
    1.  PM 進入「利害關係人」頁籤，新增一個新的利害關係人群體。
    2.  為該群體定義主要關注議題與影響力權重。
    3.  切換至「供應商」頁籤，篩選出 ESG 評分低於平均的供應商。
    4.  點擊某個供應商，查看其詳細的 ESG 評分報告、風險分級與承諾書狀態。
    5.  發送永續承諾書簽署提醒給特定供應商。

3.  UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
    桌面版佈局： 採用「左右雙欄佈局」或「多頁籤式佈局」。左側為利害關係人/供應商列表，右側為詳細資訊面板或數據圖表。
    核心液態玻璃元件： BrandDataTable (利害關係人/供應商列表，支援篩選排序)、BrandScoreCard (顯示供應商 ESG 評分與風險等級)、BrandRadarChart (利害關係人關注議題雷達圖)、BrandStatusPill (承諾書狀態標籤)。
    行動端適配 (RWD)： < 768px 時，列表自動轉化為「堆疊式資料卡片」，詳細資訊面板則以全螢幕模式呈現。篩選器收合為底部 Tab 或懸浮 FAB。

4.  核心邏輯與 5T 協定 (Logic & 5T Protocol)
    資料流向：
    *   `suppliers`：儲存供應商基本資訊、ESG 評分、風險分級、承諾書狀態、在地採購狀態。
    *   `stakeholders`：儲存利害關係人群體類型、主要關注議題、影響力權重、參與頻率。
    *   `finance_roi`：儲存與外部合作相關的投入成本、年度節省金額、回收期。
    這些資料會相互關聯，例如供應商的 ESG 評分可能影響其風險分級，並最終影響企業的財務 ROI。
    5T 實踐點：
    [T1 Tangible 具體]： 將抽象的「供應商 ESG 表現」具體化為可量化的評分、風險等級和承諾書狀態，讓使用者能直觀理解其對企業的潛在影響。
    [T3 Trackable 追蹤]： 每個供應商的 ESG 評分歷史、風險等級變化、承諾書簽署進度，以及與利害關係人的互動紀錄，都可被追溯，確保決策過程有據可查。
    [T4 Transparent 透明]： 供應商的 ESG 評分標準、風險評估模型以及利害關係人影響力權重的計算方式，都可在系統中查閱，確保評估過程的公開透明。

5.  功能項目解說和使用技術 (Features & Tech Stack)
    利害關係人管理： 支援新增/編輯利害關係人群體，定義其關注議題、影響力權重、參與頻率，並可記錄互動歷史。技術使用 React Hook Form 進行表單管理，搭配多選標籤輸入框。
    供應商 ESG 評估與風險分級： 根據預設或自定義的 ESG 指標對供應商進行評分，並自動計算風險等級。技術使用後端 Python 服務執行評分模型，前端使用 BrandScoreCard 視覺化呈現。
    承諾書狀態追蹤與提醒： 追蹤供應商永續承諾書的簽署狀態，並可發送自動提醒。技術使用 Supabase Realtime 監聽狀態變更，後端 Cron Job 服務發送郵件通知。
    財務 ROI 分析： 針對與外部合作相關的 ESG 專案，計算投入成本、年度節省金額與回收期。技術使用 Recharts 呈現 ROI 趨勢圖，後端服務進行財務數據計算。

6.  品質達標與驗收紅線 (QA Red Lines)
    🚨 UI 跑版紅線： 供應商列表或利害關係人列表在數據量大時，表格必須正常滾動，不可出現內容溢出或佈局錯亂。供應商名稱或地址過長時必須觸發 text-overflow: ellipsis。
    🚨 邏輯/體驗紅線： 當篩選條件導致無數據時，必須顯示友善的「無數據」提示。發送承諾書提醒時，必須確保郵件內容正確且能成功送達目標供應商。供應商 ESG 評分計算結果必須與輸入數據邏輯一致，不可出現計算錯誤。

## 永續情報中心 [Sustainability Intelligence Hub]
路徑： /intelligence/hub | 權限： PM, CSO, Analyst | 所屬旅程： IV. AI 賦能與撰寫

1.  模組定位 (Core Purpose)
    永續情報中心是企業獲取最新 ESG 法規、產業動態、標竿資訊與標準庫的知識中樞，透過 AI 智能分析與個人化推薦，賦能使用者掌握永續趨勢，制定前瞻性策略。

2.  客戶旅程與 UX 體驗 (Customer Journey & UX)
    目標對象與痛點： 永續專案經理 (PM)、永續長 (CSO)、分析師。他們需要隨時掌握不斷變化的 ESG 法規與市場趨勢，但資訊來源繁雜、更新不及時，難以快速篩選出與企業相關的關鍵情報。
    體驗高光時刻 (Aha Moment)： 當 PM 進入情報中心，看到系統自動根據企業產業別與關注議題，推薦了「歐盟碳邊境調整機制 (CBAM) 最新進展」的摘要與衝擊分析，並標示出「高衝擊」標籤。PM 瞬間感受到「不用自己大海撈針，關鍵情報自動送上門」的價值。
    操作軌跡：
    1.  PM 進入「永續情報中心」。
    2.  瀏覽「法規動態」頁籤，查看與企業相關的最新 ESG 法規更新。
    3.  點擊某條法規，查看其詳細解讀、衝擊等級與建議行動。
    4.  切換至「產業標竿」頁籤，比較企業在 E/S/G 各面向與同業的表現。
    5.  在「標準庫」中搜尋 GRI 305 指標，查看其詳細要求與參考範例。

3.  UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
    桌面版佈局： 採用「卡片式資訊流 (Card-based Information Feed)」佈局，頂部為搜尋與篩選器，下方為可滾動的情報卡片。右側可選配「AI 摘要與建議」側邊欄。
    核心液態玻璃元件： BrandIntelligenceCard (顯示情報標題、來源、日期、衝擊等級標籤)、BrandSearchInput (帶有自動完成功能的搜尋框)、BrandFilterChips (篩選標籤)、BrandComparisonChart (產業標竿比較圖)。
    行動端適配 (RWD)： < 768px 時，情報卡片自動堆疊成單欄佈局，搜尋與篩選器收合為頂部固定欄。AI 摘要側邊欄轉為底部彈出式面板或全螢幕模式。

4.  核心邏輯與 5T 協定 (Logic & 5T Protocol)
    資料流向：
    *   `intelligence`：儲存 ESG 法規動態、碳政策、產業新聞，包含標題、內容摘要、來源、發佈日期、衝擊等級標籤。
    *   `benchmarks`：儲存產業標竿企業的 E/S/G 分數，用於比較分析。
    *   `standards_library`：儲存 GRI, ISSB, TCFD, SASB 等永續標準的詳細條款與解釋。
    *   `net_zero_milestones`：儲存企業中長期減碳里程碑與 SBTi 目標設定。
    AI 服務會定期抓取外部情報，進行分析並寫入 `intelligence` 表。
    5T 實踐點：
    [T1 Tangible 具體]： 將複雜的法規條文、產業新聞摘要為易於理解的情報卡片，並透過「衝擊等級標籤」具體化其對企業的影響。
    [T4 Transparent 透明]： 每一條情報都清晰標示其原始來源（如：歐盟官方公報、某知名媒體），確保資訊的透明度與可信度。AI 摘要也會標註其生成依據。
    [T5 Trustworthy 信任]： 系統會定期比對外部法規資料庫，確保 `standards_library` 中的標準條款是最新的，並對過期或失效的法規進行標示，防止使用者依據過時資訊做出決策。

5.  功能項目解說和使用技術 (Features & Tech Stack)
    AI 智能情報推薦與摘要： 根據企業設定的產業別、關注議題，AI 自動推薦相關情報，並提供關鍵摘要與衝擊分析。技術使用 Genkit 搭配 Gemini 2.0 Pro 模型進行內容分析與摘要，結合 Supabase Edge Functions 實現即時推薦。
    多維度情報篩選與搜尋： 支援按法規類型、衝擊等級、發佈日期、關鍵字等進行篩選與全文檢索。技術使用 Supabase PostgreSQL 的 `tsvector` 進行全文檢索，React Hook Form 處理篩選狀態。
    產業標竿比較： 視覺化呈現企業與同業在 E/S/G 各面向的表現差異。技術使用 Recharts 或 D3.js 繪製雷達圖或條形圖，後端服務提供標竿數據。
    永續標準庫查詢： 提供可搜尋的永續標準條款資料庫，包含詳細解釋與應用範例。技術使用 Algolia 或 MeiliSearch 進行快速全文檢索，前端使用 React Markdown 渲染標準內容。

6.  品質達標與驗收紅線 (QA Red Lines)
    🚨 UI 跑版紅線： 情報卡片內容（標題、摘要）過長時必須觸發 text-overflow: ellipsis，配合 Tooltip 檢視完整內容，不可撐破卡片容器。產業標竿比較圖表在數據點過多時，必須保持可讀性，軸標籤不可重疊。
    🚨 邏輯/體驗紅線： AI 推薦的情報必須與企業設定的偏好高度相關，不可出現大量無關資訊。法規更新必須在官方發佈後 24 小時內同步至系統。搜尋功能必須在 1 秒內返回結果，並提供相關性排序。

## AI 助手設定與監控 [AI Assistant Configuration]
路徑： /admin/ai-config | 權限： ADMIN, PM | 所屬旅程： IV. AI 賦能與撰寫

1.  模組定位 (Core Purpose)
    AI 助手設定與監控模組提供對 ESG GO 系統中 AI 助手（如 SustainWrite 的 AI 擴寫、Hermes Swarm 的任務調度）進行配置、訓練與監控的介面，確保 AI 服務的效能、合規性與道德一致性。

2.  客戶旅程與 UX 體驗 (Customer Journey & UX)
    目標對象與痛點： 系統管理員 (ADMIN)、永續專案經理 (PM)。ADMIN 需要確保 AI 服務穩定運行且符合成本效益；PM 則希望微調 AI 的回應風格與知識庫，使其更符合企業文化與永續策略，但缺乏直觀的配置工具。
    體驗高光時刻 (Aha Moment)： 當 PM 調整了 AI 助手的「道德 DNA 權重」滑桿，並在測試區輸入問題時，AI 的回應風格立刻從「保守嚴謹」轉變為「積極創新」。PM 瞬間感受到「AI 助手可以被塑造成企業專屬夥伴」的強大賦能感。
    操作軌跡：
    1.  ADMIN 進入「AI 助手設定與監控」頁面。
    2.  點擊「Personas / Digital Twin」頁籤，選擇 SustainWrite 的 AI 助手。
    3.  調整其「SPIRIT 人格 System Prompts」與「道德 DNA 權重」。
    4.  在測試區輸入一段文字，觀察 AI 的擴寫結果與風格。
    5.  切換至「Hermes Swarm」頁籤，監控各 Agent 的任務分配與執行狀態。

3.  UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
    桌面版佈局： 採用「左右雙欄佈局」。左側為 AI 助手列表與頁籤導航，右側為選定 AI 助手的詳細配置面板與即時監控圖表。
    核心液態玻璃元件： BrandSlider (用於調整權重與參數)、BrandCodeEditor (用於編輯 System Prompts)、BrandRealtimeChart (顯示 Agent 狀態與任務分配的即時圖表)、BrandToggleSwitch (開關 AI 功能)。
    行動端適配 (RWD)： < 768px 時，左右雙欄自動堆疊。配置面板以全螢幕模式呈現，監控圖表則提供點擊放大查看詳情的功能。

4.  核心邏輯與 5T 協定 (Logic & 5T Protocol)
    資料流向：
    *   `personas_digital_twin`：儲存 AI 助手的 SPIRIT 人格 System Prompts、道德 DNA 權重、知識庫配置。
    *   `hermes_swarm`：儲存各 AI Agent 的狀態、任務分配、非同步工作佇列。
    *   `connectors`：儲存 Supabase, Gemini, ERP 等外部服務的連接狀態與配置。
    所有配置變更都會寫入對應資料表，並即時影響 AI 服務行為。AI 服務的運行狀態會持續更新 `hermes_swarm` 表。
    5T 實踐點：
    [T2 Traceable 追溯]： 所有 AI 助手的配置變更（如 System Prompts 的修改、權重調整）都會被記錄在 Audit Log 中，包含操作者、時間與變更內容，確保 AI 行為的可追溯性。
    [T4 Transparent 透明]： AI 助手的核心 System Prompts 和道德 DNA 權重配置對授權使用者公開，讓使用者理解 AI 行為的底層邏輯，並可進行微調。
    [T5 Trustworthy 信任]： 系統會持續監控 AI 模型的輸出，若檢測到任何「幻覺 (Hallucination)」或違背 ESG 標準的內容，會立即觸發警報並記錄，確保 AI 輸出的可信度。

5.  功能項目解說和使用技術 (Features & Tech Stack)
    AI 人格與提示詞配置： 允許 ADMIN/PM 編輯 AI 助手的 System Prompts，定義其角色、語氣和行為準則。技術使用 Monaco Editor 或 CodeMirror 實現代碼編輯器，搭配 React Hook Form 進行表單提交。
    道德 DNA 權重調整： 透過滑桿調整 AI 在不同道德維度（如：環境保護、社會公平、經濟效益）的權重，影響其決策偏好。技術使用 BrandSlider 元件，結合後端 AI 服務的權重計算邏輯。
    Hermes Swarm 任務監控： 實時監控各 AI Agent 的運行狀態、任務佇列、資源使用情況。技術使用 WebSocket 或 SSE 接收後端 AI 服務的實時數據，Recharts 繪製實時折線圖或儀表盤。
    外部服務連接器管理： 配置與監控 Supabase, Gemini, ERP 等外部服務的連接狀態，進行 API Key 管理與權限設定。技術使用 React Hook Form 處理連接器配置，後端服務進行連接測試與狀態監控。

6.  品質達標與驗收紅線 (QA Red Lines)
    🚨 UI 跑版紅線： 代碼編輯器在內容過多時必須正常滾動，不可撐破容器。實時監控圖表在數據量大時，軸標籤與圖例必須保持清晰可讀，不可重疊。
    🚨 邏輯/體驗紅線： AI 助手配置變更後，其行為必須在 5 秒內生效。當外部服務連接失敗時，必須即時顯示錯誤提示，並提供重試或排查指引。AI 監控數據必須在 3 秒內更新，確保實時性。若 AI 服務因配置錯誤導致崩潰，必須自動回滾至上一個穩定版本。
