ESG GO 系統核心架構與資料設計 ESG GO Core Architecture & Data Design
路徑： /system/architecture | 權限： ADMIN, DEV, PM | 所屬旅程： I. 系統基礎與平台治理

1. 模組定位 (Core Purpose)
本文件旨在定義 ESG GO 系統的整體技術架構、資料設計原則與核心技術棧，確保平台具備高可靠性、可擴展性、數據完整性與企業級安全性，為所有上層應用提供穩固基石。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 系統架構師、前後端工程師與產品經理。他們需要一個清晰、統一的技術藍圖，以避免技術債、確保模組間的協同效率，並在面對快速變化的業務需求時，能快速且安全地迭代。
體驗高光時刻 (Aha Moment)： 當開發團隊在遵循本架構規範下，成功將新功能模組無縫整合至現有系統，並在壓力測試中展現卓越的性能與穩定性時，所有成員都感受到「這套架構真的能撐起我們的野心」的信心。
操作軌跡：
1. 新進開發者閱讀此文件，快速理解系統的技術選型與設計哲學。
2. 產品經理在規劃新功能時，參考此文件評估技術可行性與潛在風險。
3. 架構師在設計複雜模組時，依循此文件確保資料流向與安全協定符合標準。
4. QA 團隊依據此文件定義的品質紅線，進行系統級的穩定性與安全性測試。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 系統整體採用統一的 `ClientLayout` 框架，提供標準化的側邊導航欄、頂部操作區與內容主體區。所有頁面均基於 12 欄響應式網格系統 (Responsive Grid System) 進行佈局，確保視覺一致性與內容對齊。
核心液態玻璃元件： 全系統統一使用 BrandDesignSystem 定義的原子與分子元件，例如：BrandButton, BrandInput, BrandModal, BrandTable, BrandNavigation, BrandLayout。特別是 `glassBg` (白/深藍磨砂) 作為背景層次感的基礎。
行動端適配 (RWD)：
*   **斷點策略：** 統一採用 `sm: 640px`, `md: 768px`, `lg: 1024px`, `xl: 1280px` 的 Tailwind CSS 斷點策略。
*   **導航模式：** `< 768px` 時，桌面端側邊欄自動收合為「漢堡選單」，並在底部顯示「行動端底部導航 (Bottom Tab Navigation)」，提供核心功能的快速切換。
*   **內容適配：** 確保所有內容區塊在不同螢幕尺寸下均能自動調整，避免水平滾動條，並優化 Safe Area 顯示，防止內容被系統 UI 遮擋。表格在行動端應自動轉化為堆疊式卡片或可滾動視圖。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向：
*   **單一事實來源 (SSOT)：** 以 Supabase PostgreSQL 作為所有正式資料的單一事實來源與結構映射目標。
*   **接口層統一：** 所有前端與後端服務統一透過 `lib/db.ts` 作為 Supabase 的接口層，確保資料操作的一致性與安全性。
*   **資料表映射：** 前端 TypeScript Interface 與 Mock Array 僅作為開發階段的 SEED，所有邏輯資料表設計必須嚴格對應後端關聯式資料表結構，降低前後端映射落差。
*   **數據一致性：** 強化模組間的資料同步能力，確保全系統欄位定義與資料來源一致，避免數據孤島與冗餘。
*   **雲端協同：** 平台整合 BlueCC Hybrid Control Plane，可根據工作負載進行本地與雲端資料的協同處理與儲存，確保高可用性與低延遲。
5T 實踐點：
*   **[T1 Tangible 具體]：** 所有 ESG 數據與其關聯的佐證文件，都必須有明確的資料模型與儲存路徑，將抽象的報告內容具體化為可查詢、可驗證的資料實體。例如，GRI 指標的填報數據直接映射至 `report_data` 表的特定欄位。
*   **[T2 Traceable 追溯]：** 透過「Audit Logs」機制，所有關鍵數據的生命週期變更（新增、修改、刪除、狀態變更）都將被記錄，包含操作者、時間戳、變更內容與 IP 地址，確保數據的可追溯性。
*   **[T3 Transparent 透明]：** 透過「AI Engine (Gemini 2.0 + OmniHermes + Genkit)」生成的內容，必須標示其推理來源（內部知識庫、外部法規或數據），並提供可供查閱的原始依據，確保 AI 輔助的透明度。
*   **[T4 Trustworthy 信任]：**
    *   **數據完整性：** 重要紀錄與關鍵文件上傳時，使用 SHA-256 進行加密封印，生成唯一的雜湊值。任何對原始檔案的微小修改都將導致雜湊值不匹配，從而觸發警報，確保數據的不可篡改性。
    *   **零知識證明 (ZKP)：** 關鍵文件支援 ZKP 驗證狀態，允許第三方在不揭露原始數據的情況下，驗證數據的真實性與合規性。
    *   **權限控制：** 敏感資料透過 Supabase 的 Row Level Security (RLS) 進行精細化保護，確保只有具備相應權限的使用者才能存取特定資料行。

5. 功能項目解說和使用技術 (Features & Tech Stack)
*   **前端應用框架 (Frontend Application Framework)：**
    *   **行為：** 提供高效能、響應式的使用者介面，支援伺服器端渲染 (SSR) 與靜態網站生成 (SSG)，優化 SEO 與首次內容繪製 (FCP)。
    *   **技術：** Next.js 15 (App Router), React 19, TypeScript, Tailwind CSS, Zustand (狀態管理)。
*   **後端與資料庫 (Backend & Database)：**
    *   **行為：** 提供穩定的資料儲存、查詢與 API 服務，支援即時資料同步與複雜的關聯式查詢。
    *   **技術：** Supabase PostgreSQL (作為主資料庫與認證服務), Supabase Realtime (即時資料同步), Supabase Storage (檔案儲存)。
*   **雲端控制與混合調度 (Cloud Control & Hybrid Dispatcher)：**
    *   **行為：** 根據工作負載與成本效益，智能調度計算資源於本地與雲端環境，確保系統彈性與高可用性。
    *   **技術：** BlueCC API Bridge (統一 API 接口), Hybrid Dispatcher (工作負載調度引擎)。
*   **AI 賦能引擎 (AI Engine)：**
    *   **行為：** 提供生成式 AI 內容撰寫、合規性檢查、數據分析與智能輔助功能。
    *   **技術：** Google Gemini 2.0 Pro (核心 LLM), OmniHermes (AI 內容審核與校準), Genkit (AI 工作流編排框架), Server-Sent Events (SSE) (實現串流響應)。
*   **數據完整性與安全層 (Integrity & Security Layer)：**
    *   **行為：** 確保數據的真實性、不可篡改性與存取安全性。
    *   **技術：** SHA-256 (數據雜湊), ZKP (零知識證明), Audit Logs (審計日誌), Supabase Row Level Security (RLS), Web Crypto API (前端雜湊運算)。
*   **通知與郵件服務 (Email / Notification Service)：**
    *   **行為：** 提供可靠的郵件發送與系統通知服務。
    *   **技術：** Resend API。
*   **API 連接器 (API Connectors)：**
    *   **行為：** 實現與第三方服務（如 ERP 系統、IoT 數據源）的安全連接與數據交換。
    *   **技術：** OAuth 2.0, JWT, API Gateway。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線：
*   **全域佈局崩潰：** 任何頁面在任何標準斷點下，出現側邊欄或底部導航重疊、內容區塊溢出、或主要元件被遮擋的情況。
*   **文字溢出：** 任何文字內容（特別是動態生成或用戶輸入的長字串）未觸發 `text-overflow: ellipsis` 而撐開容器，導致頁面出現 X 軸滾動條。
*   **Z-index 錯誤：** 模態框 (Modal)、下拉選單 (Dropdown) 或懸浮提示 (Tooltip) 被其他頁面元素遮擋，導致無法正常互動。
*   **響應式失效：** 在行動端視圖下，表格未正確轉化為堆疊式卡片，或互動式圖表未提供橫向全螢幕選項，導致操作困難。
🚨 邏輯/體驗紅線：
*   **數據完整性破壞：** 任何上傳的關鍵文件，其 SHA-256 Hash 值與資料庫記錄不符，但系統未發出警報或阻斷操作。
*   **RLS 繞過：** 任何角色在未經授權的情況下，成功存取或修改其權限範圍外的敏感資料。
*   **AI 幻覺未攔截：** AI 生成的內容包含明顯違背 ESG 標準、事實錯誤或有害的幻覺字眼，且未被 OmniHermes 或其他機制有效攔截、標示或修正。
*   **斷網處理不當：** 系統在斷網情況下，未平滑切換至離線模式（如 IndexedDB 暫存），或彈出未經處理的 Error 500 白畫面，導致用戶數據丟失或恐慌。
*   **性能瓶頸：** 任何核心 API 響應時間持續超過 3 秒 (在正常負載下)，或頁面首次內容繪製 (FCP) 超過 2 秒，影響用戶體驗。
*   **審計日誌缺失：** 任何關鍵數據的生命週期變更未被完整記錄於 Audit Logs 中，導致無法追溯操作歷史。
