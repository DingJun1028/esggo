## 平台功能頁面 (依據「功能框架建立」階段的六大領域改寫)

### 核心指揮中心 Core Command Center
路徑： `/dashboard` | 權限： `ALL_USERS` | 所屬旅程： `I. 總覽與策略`

#### 1. 模組定位 (Core Purpose)
核心指揮中心是 ESG GO 平台的入口頁面，提供企業 ESG 績效的即時總覽、關鍵指標 (KPI) 追蹤與快速任務導航，幫助使用者迅速掌握平台狀態與優先處理事項。

#### 2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
- **目標對象與痛點**： 企業高階主管 (CSO, CEO) 或永續專案經理 (PM)。他們需要快速了解企業的 ESG 表現，但往往被分散在不同系統的數據淹沒，難以一眼看出重點與潛在風險。
- **體驗高光時刻 (Aha Moment)**： 當主管登入後，看到一個清晰的「ESG 健康分數」與趨勢圖，並能直接點擊異常指標進入詳情頁面，瞬間感受到「一切盡在掌握」的安心感。
- **操作軌跡**：
  1. 登入平台後，自動進入核心指揮中心。
  2. 快速瀏覽「碳排放趨勢」、「員工滿意度」等核心 KPI。
  3. 點擊「待辦任務」區塊，查看是否有需要處理的數據填報或報告審核。
  4. 透過「快速連結」進入常用功能，如 SustainWrite 編輯器。

#### 3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
- **桌面版佈局**： 採用 12 欄 Bento Grid 佈局，頂部為「全局概覽卡片」，下方為可自定義的「KPI 儀表板小部件」與「任務列表」。
- **核心液態玻璃元件**： 使用 `BrandKPIWidget` (帶有趨勢箭頭與 Liquid Glass 背景)、`BrandTaskCard` (顯示任務進度與截止日期)。
- **行動端適配 (RWD)**： `< 768px` 時，Bento Grid 自動堆疊為單欄，KPI 小部件轉為可左右滑動的卡片列表，底部固定顯示「快速導航 Tab」。

#### 4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
- **資料流向**： KPI 數據從 `esg_metrics`、`report_status`、`task_management` 等多個資料表聚合，經由後端 API 實時計算後展示。點擊任務會導向對應的功能頁面。
- **5T 實踐點**：
  - **[T1 Tangible 具體]**： 將複雜的 ESG 數據轉化為直觀的圖表、進度條和健康分數，讓抽象的績效變得可感知。
  - **[T4 Transparent 透明]**： 每個 KPI 小部件都提供「數據來源」的 Tooltip，點擊可追溯到原始數據輸入頁面或佐證文件。

#### 5. 功能項目解說和使用技術 (Features & Tech Stack)
- **實時 KPI 儀表板**： 聚合多源數據，提供可配置的圖表展示。技術使用 React Query 進行數據緩存與刷新，結合 Recharts 或 ECharts 進行圖表渲染。
- **個性化任務與通知**： 根據使用者角色和權限，展示相關的待辦事項和系統通知。技術使用 Supabase Realtime 或 WebSocket 實現實時通知推送，並結合 `react-toastify` 顯示彈出訊息。
- **全局搜索功能**： 允許使用者快速搜索平台內的任何功能、報告或數據。技術使用 Algolia 或 Fuse.js 進行前端模糊搜索，後端則透過 PostgreSQL `tsvector` 實現全文檢索。

#### 6. 品質達標與驗收紅線 (QA Red Lines)
- 🚨 **UI 跑版紅線**： 當 KPI 數據為負值或極大值時，趨勢圖的 Y 軸範圍必須自動調整，不可出現數據點超出圖表邊界的情況。
- 🚨 **邏輯/體驗紅線**： 任何 KPI 數據更新後，儀表板上的相關圖表必須在 5 秒內刷新，確保數據的即時性。若後端 API 響應時間過長，必須顯示骨架屏 (Skeleton Loader) 而非空白頁面。

### E-S-G 數據模組 ESG Data Module
路徑： `/esg-data/input` | 權限： `PM, DATA_ENTRY` | 所屬旅程： `II. 數據收集與管理`

#### 1. 模組定位 (Core Purpose)
E-S-G 數據模組是企業所有 ESG 相關數據的收集、輸入、驗證與管理中心，確保數據的準確性、完整性與可追溯性，為後續報告撰寫與分析提供堅實基礎。

#### 2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
- **目標對象與痛點**： 各部門數據填報窗口 (如：人資、財務、環安衛)。他們常面臨數據格式不統一、填報流程複雜、缺乏即時驗證導致錯誤頻繁，以及數據來源難以追溯的問題。
- **體驗高光時刻 (Aha Moment)**： 當填報者輸入「用水量」數據後，系統立即彈出「與去年同期相比異常增加 20%，請確認或提供說明」的提示，並自動建議關聯「水費單」作為佐證。這種智能預警與引導，讓填報者感到「數據輸入不再是盲填，而是有智慧的協作」。
- **操作軌跡**：
  1. 填報者從左側導航選擇「環境數據 > 用水管理」。
  2. 選擇填報月份，輸入各廠區的用水量數據。
  3. 系統自動進行數據驗證與異常提示。
  4. 上傳相關水費單至證據金庫，並將其關聯至該筆數據。
  5. 確認無誤後，點擊「提交審核」。

#### 3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
- **桌面版佈局**： 採用「左側導航樹」與「右側內容表單區」的雙欄佈局。表單區內部為多步驟表單或分頁式數據輸入表格。
- **核心液態玻璃元件**： 使用 `BrandInputGroup` (帶有單位與提示訊息)、`BrandFileUploadZone` (整合證據金庫上傳功能)、`BrandDataGrid` (可編輯的數據表格)。
- **行動端適配 (RWD)**： `< 768px` 時，左側導航樹收合為「漢堡選單」，右側表單區滿版顯示。複雜的數據表格自動轉化為「堆疊式卡片列表」，每張卡片代表一筆數據，點擊可展開編輯。

#### 4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
- **資料流向**： 數據輸入後，經前端驗證與後端業務邏輯處理，存入 `esg_metrics` 資料表。相關佐證文件 Hash 存入 `evidence_vault`，並建立 `esg_metrics` 與 `evidence_vault` 之間的關聯。
- **5T 實踐點**：
  - **[T2 Traceable 追溯]**： 每一筆輸入的數據都記錄其「填報人」、「填報時間」、「修改歷史」與「關聯佐證文件 ID」，確保數據來源與變更歷程可追溯。
  - **[T5 Trustworthy 信任]**： 數據輸入時，若關聯佐證文件，系統會自動比對該文件的 SHA-256 Hash 值，確保文件未被篡改。

#### 5. 功能項目解說和使用技術 (Features & Tech Stack)
- **智能數據驗證與預警**： 根據預設規則（如：範圍、趨勢、與歷史數據比較）對輸入數據進行實時驗證，並提供異常提示。技術使用 `react-hook-form` 進行前端表單驗證，後端則使用 Python `Pandas` 進行數據清洗與異常檢測。
- **佐證文件一鍵關聯**： 允許使用者直接從數據輸入頁面將文件上傳至證據金庫並建立關聯。技術使用 `react-dropzone` 實現拖曳上傳，並透過 Supabase Storage API 進行文件管理，Web Crypto API 計算 Hash。
- **多維度數據篩選與匯出**： 提供按時間、部門、指標等條件篩選數據，並支持匯出為 Excel/CSV 格式。技術使用 `TanStack Table` 實現高性能數據表格，並結合 `xlsx` 庫進行前端匯出。

#### 6. 品質達標與驗收紅線 (QA Red Lines)
- 🚨 **UI 跑版紅線**： 當數據輸入表單包含大量欄位時，表單的滾動條必須正常出現，且表單標籤與輸入框的對齊必須保持一致，不可出現錯位或溢出。
- 🚨 **邏輯/體驗紅線**： 若使用者在未保存數據的情況下嘗試離開頁面，必須彈出「您有未保存的變更，確定要離開嗎？」的提示，防止數據意外丟失。

### 治理與確信 Governance & Assurance
路徑： `/governance/audit` | 權限： `CSO, AUDITOR, ADMIN` | 所屬旅程： `III. 治理與風險`

#### 1. 模組定位 (Core Purpose)
治理與確信模組提供企業內部 ESG 政策、風險管理、合規性審查與稽核追蹤的功能，確保企業的 ESG 實踐符合內外部標準，並為第三方確信提供透明、可驗證的依據。

#### 2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
- **目標對象與痛點**： 永續長 (CSO)、內部稽核員、外部確信機構。他們需要評估企業 ESG 政策的執行情況、識別潛在風險、追蹤合規進度，但往往面臨文件分散、審核流程不透明、稽核證據難以收集的問題。
- **體驗高光時刻 (Aha Moment)**： 當稽核員在系統中點擊某項政策，不僅能看到政策內容，還能直接看到該政策下所有相關的「執行紀錄」、「負責人」與「佐證文件連結」，並能一鍵生成稽核報告。這種「一站式透明化」讓稽核員感到「審核效率大幅提升，信任感油然而生」。
- **操作軌跡**：
  1. 稽核員登入後，進入「治理與確信」模組。
  2. 選擇「政策合規性審查」，瀏覽各項 ESG 政策列表。
  3. 點擊「反貪腐政策」，查看其執行狀態與相關風險評估。
  4. 閱系統自動彙整的「政策執行紀錄」與「佐證文件」，並在頁面中直接添加稽核意見。
  5. 點擊「生成稽核報告」，匯出為 PDF。

#### 3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
- **桌面版佈局**： 採用「左側政策/風險樹狀導航」與「右側內容詳情頁」的佈局。詳情頁內部包含「政策文本」、「執行進度儀表板」與「相關文件列表」。
- **核心液態玻璃元件**： 使用 `BrandPolicyCard` (顯示政策狀態與風險等級)、`BrandAuditLogTable` (具備時間戳與操作人資訊的表格)、`BrandRiskMatrix` (互動式風險矩陣圖)。
- **行動端適配 (RWD)**： `< 768px` 時，左側導航收合為「漢堡選單」，右側內容區滿版。風險矩陣圖自動轉化為「風險列表」，點擊可展開詳細資訊。

#### 4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
- **資料流向**： 政策內容存於 `esg_policies` 表，風險評估結果存於 `risk_assessments` 表，稽核紀錄存於 `audit_logs` 表。所有相關文件連結至 `evidence_vault`。
- **5T 實踐點**：
  - **[T3 Trackable 追蹤]**： 系統自動記錄所有政策的創建、修改、審核與發佈歷史，以及每次稽核的結果與意見，確保所有治理活動都有完整的軌跡。
  - **[T5 Trustworthy 信任]**： 稽核報告生成時，會自動嵌入所有引用佐證文件的 SHA-256 Hash 碼，確保報告內容與原始證據的不可篡改性。

#### 5. 功能項目解說和使用技術 (Features & Tech Stack)
- **政策與風險管理**： 提供政策文件上傳、版本控制、審批流程配置與風險評估工具。技術使用 Supabase Storage 進行文件管理，`react-flow` 或 `bpmn-js` 實現流程可視化。
- **自動化合規性檢查**： 根據預設的法規與標準（如 GRI, ISSB），自動比對企業數據與政策，識別潛在的合規性缺口。技術使用 Genkit 搭配 Gemini 模型進行文本分析與比對，並結合規則引擎 (e.g., `json-rules-engine`)。
- **稽核報告生成器**： 允許使用者自定義報告模板，自動匯總稽核發現、建議與佐證連結。技術使用 `react-pdf` 或 `jsPDF` 進行前端 PDF 生成，後端則可使用 `Puppeteer` 進行無頭瀏覽器渲染。

#### 6. 品質達標與驗收紅線 (QA Red Lines)
- 🚨 **UI 跑版紅線**： 當政策文本內容過長時，編輯器或顯示區的滾動條必須正常工作，且頁面佈局不可因內容過多而變形。
- 🚨 **邏輯/體驗紅線**： 稽核員在提交稽核意見後，該意見必須立即在相關政策頁面中可見，且不可被非授權用戶修改。若嘗試修改已「封存」的稽核報告，必須彈出明確的權限不足提示。

### 洞察與發佈 Insights & Publishing
路徑： `/reports/publish` | 權限： `PM, CSO, ADMIN` | 所屬旅程： `IV. 洞察與發佈`

#### 1. 模組定位 (Core Purpose)
洞察與發佈模組是企業 ESG 數據分析、績效洞察與永續報告發佈的核心平台，將原始數據轉化為有意義的商業洞察，並支持多格式、多標準的報告生成與對外溝通。

#### 2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
- **目標對象與痛點**： 永續專案經理 (PM)、永續長 (CSO)、投資者關係部門。他們需要從海量數據中提煉關鍵洞察、製作專業的永續報告，但往往耗費大量時間在數據整理、圖表製作與格式調整上，且難以確保報告內容的合規性與可信度。
- **體驗高光時刻 (Aha Moment)**： 當 PM 點擊「生成 GRI 報告」後，系統在數秒內自動匯總所有相關數據、圖表與撰寫內容，並生成一份符合 GRI 標準的 PDF 報告草稿。PM 看到報告中所有數據都帶有「可追溯至原始佐證」的標記時，瞬間感受到「報告撰寫不再是苦差事，而是高效且可信的成果」。
- **操作軌跡**：
  1. PM 進入「洞察與發佈」模組，選擇「報告生成」。
  2. 選擇報告類型（如：GRI 永續報告、TCFD 氣候報告）與報告週期。
  3. 系統自動匯總相關數據與撰寫內容，並展示報告預覽。
  4. PM 在預覽模式下進行最終審閱與微調。
  5. 點擊「發佈」，選擇發佈格式（PDF, 網頁版）與對外分享選項。

#### 3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
- **桌面版佈局**： 採用「左側報告章節導航」與「右側報告預覽/編輯區」的雙欄佈局。預覽區支持所見即所得編輯與數據圖表互動。
- **核心液態玻璃元件**： 使用 `BrandReportViewer` (支持 PDF 嵌入與網頁渲染)、`BrandChartGallery` (可拖曳配置的圖表庫)、`BrandPublishButton` (帶有發佈選項的按鈕)。
- **行動端適配 (RWD)**： `< 768px` 時，左側導航收合為「漢堡選單」，右側報告預覽區滿版。報告中的圖表自動適配為響應式佈局，或提供「點擊放大」功能。

#### 4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
- **資料流向**： 報告內容從 `report_drafts`、`esg_metrics`、`evidence_vault` 等多個資料表聚合。發佈後的報告存入 `published_reports` 表，並可選擇同步至外部 CDN 或公開網頁。
- **5T 實踐點**：
  - **[T4 Transparent 透明]**： 報告中所有引用的數據和關鍵聲明，都提供可點擊的連結，直接導向其在證據金庫中的原始佐證文件，確保報告內容的透明度。
  - **[T5 Trustworthy 信任]**： 發佈的報告會自動生成一個唯一的區塊鏈 Hash (或類似的防篡改標記)，確保報告內容一旦發佈即不可被篡改，提升報告的可信度。

#### 5. 功能項目解說和使用技術 (Features & Tech Stack)
- **多標準報告生成**： 支持 GRI、ISSB、TCFD 等多種國際永續報告標準的自動化生成。技術使用 Genkit 搭配 Gemini 模型進行內容生成與合規性檢查，並結合客製化模板引擎 (如 `Handlebars.js`)。
- **互動式數據圖表**： 報告中嵌入的圖表支持數據鑽取、篩選與導出。技術使用 D3.js 或 ECharts 進行高性能圖表渲染，並結合 React Context API 進行數據狀態管理。
- **多格式發佈與分享**： 支持將報告發佈為 PDF、響應式網頁、或生成可嵌入的 iframe。技術使用 `react-pdf` 或 `html2canvas` 進行 PDF 轉換，並利用 Supabase Storage 或 AWS S3 進行靜態文件託管。

#### 6. 品質達標與驗收紅線 (QA Red Lines)
- 🚨 **UI 跑版紅線**： 報告預覽模式下，任何圖表或圖片的尺寸調整都必須保持其長寬比，不可出現圖片被拉伸或壓縮變形的情況。
- 🚨 **邏輯/體驗紅線**： 報告發佈前，系統必須強制執行「合規性掃描」，若存在任何未達標的指標或未提供佐證的聲明，必須阻斷發佈並明確提示使用者進行修正。

### 學院與顧問 Academy & Consulting
路徑： `/academy` | 權限： `ALL_USERS` | 所屬旅程： `VI. 學習與支援`

#### 1. 模組定位 (Core Purpose)
學院與顧問模組是 ESG GO 平台的知識庫與學習中心，提供豐富的 ESG 學習資源、最佳實踐案例、常見問題解答，並提供專業顧問服務的預約與管理介面，賦能使用者提升 ESG 專業能力。

#### 2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
- **目標對象與痛點**： 剛接觸 ESG 的企業員工、需要深入了解特定 ESG 議題的專案經理、尋求外部專業指導的企業高管。他們常面臨 ESG 知識體系龐大、資訊碎片化、難以找到可靠學習資源，以及缺乏專業顧問支持的問題。

- **操作軌跡**：
  1. 使用者進入「學院與顧問」模組。
  2. 在搜索框中輸入感興趣的 ESG 議題。
  3. 瀏覽搜索結果中的課程、文章、案例研究。
  4. 點擊進入某個課程，觀看教學影片並完成測驗。
  5. 若有進一步需求，點擊「預約顧問」按鈕，選擇服務類型與時間。

#### 3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
- **桌面版佈局**： 採用「頂部搜索欄」與「下方卡片式內容網格」的佈局。右側可彈出「推薦顧問」側邊抽屜。
- **核心液態玻璃元件**： 使用 `BrandCourseCard` (顯示課程進度與評分)、`BrandArticleCard` (顯示文章標題與閱讀時間)、`BrandConsultantProfile` (顯示顧問專長與預約按鈕)。
- **行動端適配 (RWD)**： `< 768px` 時，卡片網格自動轉化為單欄堆疊，搜索欄固定在頂部。推薦顧問側邊抽屜轉化為底部彈出式 Modal。

#### 4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
- **資料流向**： 課程內容、文章、FAQ 存於 `academy_content` 表。顧問資料存於 `consultants` 表，預約紀錄存於 `booking_records` 表。
- **5T 實踐點**：
  - **[T1 Tangible 具體]**： 將抽象的 ESG 知識點具體化為結構化的課程、易讀的文章和可操作的工具，讓學習成果可衡量。
  - **[T4 Transparent 透明]**： 顧問的資歷、專長與過往服務評價公開透明，使用者可根據真實反饋選擇合適的顧問。

#### 5. 功能項目解說和使用技術 (Features & Tech Stack)
- **智能知識搜索**： 支持關鍵字搜索、標籤篩選與內容推薦。技術使用 Algolia 或 Elasticsearch 進行全文檢索，結合 AI 模型進行語義搜索與內容推薦。
- **互動式學習課程**： 提供影片播放、測驗、進度追蹤與證書頒發。技術使用 `react-player` 播放影片，`react-quiz` 實現測驗功能，並透過 Supabase Edge Functions 處理證書生成。
- **顧問預約與管理**： 提供顧問日程展示、在線預約、支付與會議管理。技術使用 `react-big-calendar` 顯示日程，Stripe 或綠界科技進行支付集成，並結合 Zoom 或 Google Meet API 進行會議創建。

### 系統管理 System Management
路徑： `/admin/settings` | 權限： `ADMIN` | 所屬旅程： `VII. 系統配置與維護`

#### 1. 模組定位 (Core Purpose)
系統管理模組是 ESG GO 平台的後台配置與維護中心，提供用戶管理、權限設定、系統參數配置、集成管理與數據備份等功能，確保平台的穩定運行、安全合規與高效管理。

#### 2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
- **目標對象與痛點**： 系統管理員 (ADMIN)。他們需要管理平台用戶、配置系統行為、監控系統狀態，但往往面臨操作介面複雜、權限控制粒度不足、集成配置困難，以及缺乏系統級監控與備份工具的問題。
- **體驗高光時刻 (Aha Moment)**： 當管理員需要為新入職的永續專案經理分配權限時，只需在用戶列表中點擊「新增用戶」，選擇「PM」角色，系統便自動應用預設權限組。這種「一鍵式角色分配」讓管理員感到「系統管理變得前所未有的簡單與高效」。
- **操作軌跡**：
  1. 管理員登入後，進入「系統管理」模組。
  2. 選擇「用戶與權限管理」。
  3. 點擊「新增用戶」，輸入新用戶資訊並選擇角色。
  4. 瀏覽「系統日誌」，檢查是否有異常操作或錯誤。
  5. 進入「集成管理」，配置與第三方系統（如 ERP）的數據同步。

#### 3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
- **桌面版佈局**： 採用「左側功能導航列表」與「右側內容配置區」的雙欄佈局。配置區內部多為表單、數據表格與開關元件。
- **核心液態玻璃元件**： 使用 `BrandUserTable` (顯示用戶資訊與操作按鈕)、`BrandToggleSwitch` (用於開關功能)、`BrandIntegrationCard` (顯示集成狀態與配置入口)。
- **行動端適配 (RWD)**： `< 768px` 時，左側導航收合為「漢堡選單」，右側內容區滿版。複雜的數據表格自動轉化為「堆疊式資料卡片」，每張卡片代表一個用戶或一條日誌。

#### 4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
- **資料流向**： 用戶資料存於 `auth.users` 與 `user_profiles` 表，權限配置存於 `user_roles` 與 `permissions` 表。系統日誌存於 `system_logs` 表。集成配置存於 `integrations_config` 表。
- **5T 實踐點**：
  - **[T2 Traceable 追溯]**： 系統自動記錄所有管理員的操作行為（如：新增用戶、修改權限、配置集成），包括操作時間、操作人與具體變更內容，確保所有系統級變更都有完整的追蹤紀錄。
  - **[T5 Trustworthy 信任]**： 權限管理採用最小權限原則，並透過 Supabase 的 Row Level Security (RLS) 確保數據層級的訪問安全，防止未經授權的數據讀取或修改。

#### 5. 功能項目解說和使用技術 (Features & Tech Stack)
- **精細化用戶與權限管理**： 支持多角色、多層級的權限配置，可自定義權限組。技術使用 Supabase Auth 進行用戶認證，並結合自定義的 `user_roles` 與 `permissions` 表實現 RBAC (Role-Based Access Control)。
- **系統日誌與監控**： 提供實時的系統操作日誌、錯誤日誌與性能監控儀表板。技術使用 ELK Stack (Elasticsearch, Logstash, Kibana) 或 Grafana + Prometheus 進行日誌聚合與監控可視化。
- **第三方系統集成**： 提供與企業 ERP、CRM 或其他數據源的 API 集成配置介面。技術使用 `axios` 或 `fetch` 進行 API 請求，並透過 BlueCC 混合雲調度邏輯實現安全、高效的數據同步。

#### 6. 品質達標與驗收紅線 (QA Red Lines)
- 🚨 **UI 跑版紅線**： 用戶列表或日誌表格在數據量龐大時，分頁器必須正常工作，且表格列寬自動調整，不可出現水平滾動條或內容溢出。
- 🚨 **邏輯/體驗紅線**： 任何涉及敏感操作（如：刪除用戶、重置密碼）的行為，必須彈出二次確認對話框，並要求管理員輸入密碼或進行雙因素認證，以防止誤操作。
