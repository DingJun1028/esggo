# ESG 治理中心 ESG Governance Hub [Governance]
路徑： /governance | 權限： CSO, ADMIN, PM | 所屬旅程： VI. 績效監控與持續改進

1.  **模組定位 (Core Purpose)**
    ESG 治理中心提供企業高階主管與永續團隊一個整合性的平台，用於監控 ESG 績效、管理治理風險、確保合規性，並推動永續策略的有效執行，以提升企業的永續韌性與市場信任度。

2.  **客戶旅程與 UX 體驗 (Customer Journey & UX)**
    **目標對象與痛點：** 永續長 (CSO)、董事會成員、法務合規主管。他們面臨的痛點是缺乏即時、整合的 ESG 治理數據，難以有效追蹤各項治理目標的進度，且在面對監管機構或投資人質詢時，往往需要耗費大量時間手動彙整資料以證明合規性與績效。
    **體驗高光時刻 (Aha Moment)：** 當 CSO 在董事會會議上，直接透過 ESG 治理中心展示各項治理 KPI 的即時進度，並能一鍵鑽取至相關政策文件與稽核紀錄時，不僅展現了數據透明度與專業度，更讓董事會成員感受到「所有治理資訊盡在掌握」的安心與效率。
    **操作軌跡：**
    1.  CSO 登入系統，首先檢視「治理總覽儀表板」，快速掌握董事會多元性、倫理培訓完成率、風險事件數量等關鍵指標。
    2.  發現某項合規指標呈現黃燈警示，點擊該卡片鑽取至「合規監控頁面」，查看詳細的違規事件與負責人。
    3.  在合規頁面中，直接指派任務給相關部門，並設定追蹤期限。
    4.  生成一份「季度治理報告」，供董事會審閱，報告中自動包含所有最新數據與圖表。

3.  **UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)**
    **桌面版佈局：** 採用「多卡片儀表板佈局」，左側為可收合的導航選單，頂部為全局篩選器（如時間區間、部門），中央區域則由多個可拖曳、可調整大小的 KPI 儀表板卡片組成，卡片內含圖表、進度條或數據列表。
    **核心液態玻璃元件：** `BrandKPIWidget` (顯示關鍵指標與趨勢圖)、`GovernanceRiskGauge` (環形風險評分儀表板)、`ComplianceTimeline` (時間軸式合規事件列表)、`DynamicFilterBar` (動態篩選器)。
    **行動端適配 (RWD)：** < 768px 時，左側導航自動收合為漢堡選單，頂部篩選器轉化為底部浮動按鈕 (FAB) 或可展開的抽屜式篩選器。所有儀表板卡片自動堆疊為單欄佈局，並優先顯示核心數據，複雜圖表可點擊展開全螢幕檢視。

4.  **核心邏輯與 5T 協定 (Logic & 5T Protocol)**
    **資料流向：**
    *   **輸入：** 數據來源包括：來自「人力資源模組」的員工多元性數據、來自「風險管理模組」的風險事件紀錄、來自「合規管理模組」的政策更新與審核狀態、來自「利害關係人溝通模組」的互動紀錄。
    *   **處理：** 數據經由後端服務進行聚合、計算與轉換，生成各項治理 KPI。
    *   **輸出：** 處理後的 KPI 數據實時更新至前端儀表板，並存入 `governance_metrics` 資料表，同時所有關鍵操作與數據變更會寫入 `audit_logs` 資料表。
    **5T 實踐點：**
    *   **[T1 Tangible 具體]：** 將抽象的「公司治理」概念，具體化為可量化的 KPI (如：董事會女性比例、獨立董事比例、倫理培訓完成率)，並以直觀的圖表呈現，讓管理者能清晰掌握治理現況。
    *   **[T3 Trackable 追蹤]：** 系統自動追蹤各項治理目標的達成進度（例如：設定董事會多元性目標為 30%，系統實時顯示當前進度與距離目標的差距），並記錄所有政策變更與審批流程，確保可追溯性。
    *   **[T5 Trustworthy 信任]：** 所有輸入的原始數據在進入系統前，會經過嚴格的數據驗證與清洗流程。關鍵治理數據的計算邏輯公開透明，並提供數據來源的鑽取功能，確保數據的真實性與可信度。

5.  **功能項目解說和使用技術 (Features & Tech Stack)**
    *   **治理 KPI 儀表板：** 提供多維度的治理指標展示，支援自定義卡片佈局與數據篩選。技術使用 React Grid Layout 實現卡片拖曳與調整大小，搭配 ECharts 或 Recharts 進行數據可視化渲染。
    *   **風險與合規監控：** 實時監控治理相關的風險事件與合規狀態，並提供預警機制。技術使用後端服務 (如 Node.js with NestJS) 處理事件流 (Event Streaming) 與規則引擎 (Rule Engine) 進行風險評估，並透過 WebSocket 實現前端即時通知。
    *   **政策與文件管理整合：** 允許上傳、版本控制與關聯 ESG 治理相關的政策文件、董事會會議紀錄等。技術使用 Supabase Storage 或 AWS S3 進行文件儲存，並結合 PostgreSQL 的 JSONB 欄位儲存文件元數據與版本資訊。
    *   **利害關係人溝通追蹤：** 記錄與追蹤董事會、股東、監管機構等利害關係人的溝通互動與承諾。技術使用 GraphQL API 進行數據查詢，並透過 React Hook Form 實現高效的表單輸入與管理。

6.  品質達標與驗收紅線 (QA Red Lines)
    🚨 **UI 跑版紅線：**
    *   當儀表板卡片數量過多或數據內容過長時，卡片之間必須保持預設間距，不可出現重疊或溢出父容器導致頁面出現橫向滾動條。
    *   在不同解析度下，圖表內的文字標籤與圖例必須清晰可見，不可出現文字截斷或重疊。
    🚨 **邏輯/體驗紅線：**
    *   任何治理 KPI 的計算結果必須與其底層數據源保持一致，誤差率不得超過 0.01%。若數據來源更新，儀表板上的 KPI 必須在 5 秒內完成刷新。
    *   當使用者嘗試訪問其權限範圍之外的治理數據或功能時，系統必須明確提示權限不足，而非顯示空白頁面或錯誤訊息。
    *   所有關鍵的治理數據（如董事會成員名單、政策文件）在刪除或修改前，必須彈出二次確認對話框，並在操作成功後，相關變更必須在 Audit Log 中留下不可篡改的紀錄。
