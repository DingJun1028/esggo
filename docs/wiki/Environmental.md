# Environmental 環境數據管理 [Environmental Data Management]
路徑： /environmental | 權限： PM, ESG_MANAGER, ADMIN | 所屬旅程： III. 數據採集與績效監控

1.  模組定位 (Core Purpose)
    環境數據管理模組是 ESG GO 的核心數據匯集與分析平台，旨在協助企業高效收集、計算並監控其環境足跡（如溫室氣體 Scope 1-3、能源、水資源、廢棄物），確保數據的準確性與合規性，為永續報告提供堅實基礎。

2.  客戶旅程與 UX 體驗 (Customer Journey & UX)
    目標對象與痛點： 永續數據分析師、廠區環安衛經理。他們常面臨來自不同廠區、不同格式的環境數據（電費單、水費單、廢棄物清運單），手動匯總耗時且易出錯，難以即時掌握整體環境績效，更遑論進行精確的 Scope 1-3 排放計算與追溯。
    體驗高光時刻 (Aha Moment)： 當分析師上傳一份新的電費單 PDF，系統不僅自動識別並提取關鍵數據，還在 5 秒內將其納入 GHG Scope 2 的即時計算中，並在儀表板上更新碳排放趨勢圖。分析師會驚嘆：「原來數據管理可以這麼輕鬆、這麼即時！」
    操作軌跡：
    1.  永續數據分析師登入系統，進入「環境數據管理」頁面。
    2.  點擊「上傳數據」，選擇多份來自不同廠區的電費、水費、燃料單據 PDF。
    3.  系統自動進行 OCR 識別與數據預處理，並提示待確認的異常數據點。
    4.  分析師快速審閱並修正少量錯誤，確認後數據自動入庫並觸發 GHG 計算。
    5.  切換至「環境儀表板」，檢視更新後的 Scope 1-3 排放總量與趨勢，並鑽取至特定廠區的數據細節。

3.  UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
    桌面版佈局： 採用「頂部數據總覽卡片 + 左側導航樹 + 中央數據表格/圖表」的複合佈局。頂部為可切換時間範圍的 KPI 總覽卡片，中央區域可透過 Tab 切換「原始數據列表」、「GHG 排放分析」與「能源/水/廢棄物趨勢圖」。
    核心液態玻璃元件： BrandKPIWidget (顯示總排放量、能源消耗等)、BrandDataTable (用於原始數據展示與編輯)、BrandLineChart (趨勢圖)、BrandFileUploadZone (數據上傳區)。
    行動端適配 (RWD)： < 768px 時，頂部 KPI 卡片自動堆疊，左側導航樹收合為漢堡選單。中央數據表格自動轉化為「堆疊式資料卡片 (Stacked Cards)」，僅顯示關鍵數據，並提供「點擊展開詳情」功能。圖表則自動調整為單欄佈局，並支援手勢縮放。

4.  核心邏輯與 5T 協定 (Logic & 5T Protocol)
    資料流向：
    1.  **數據採集：** 透過手動上傳 (PDF/Excel) 或 API 串接 (如智慧電錶、ERP 系統)，原始數據進入 `raw_environmental_data` 資料表。
    2.  **數據處理：** 系統對原始數據進行清洗、單位轉換與缺失值處理，並根據預設或自定義的排放因子，計算出 GHG Scope 1-3 排放量，結果寫入 `ghg_emissions`、`energy_consumption` 等專屬資料表。
    3.  **數據展示：** 儀表板與報告生成器從這些處理後的資料表中撈取數據，進行實時可視化與分析。
    5T 實踐點：
    [T1 Tangible 具體]： 將抽象的電量 (kWh)、燃料消耗 (公升) 等數據，透過 IPCC 或在地化排放因子，具體轉換為可量化的二氧化碳當量 (tCO2e)，並以直觀的圖表呈現。
    [T2 Traceable 追溯]： 每一筆計算出的 GHG 排放數據，都可追溯回其原始數據來源（如：哪份電費單、哪個 API 請求），並記錄其計算所使用的排放因子版本。
    [T3 Trackable 追蹤]： 任何數據的修改、排放因子的更新或計算方法的調整，都會在該數據點的 Audit Log 中留下詳細紀錄，包含修改者、時間與前後值。
    [T4 Transparent 透明]： 系統會明確標示所使用的排放因子來源（如：環保署、IPCC AR6），並提供連結至其官方文件，讓使用者對計算基礎一目瞭然。
    [T5 Trustworthy 信任]： 數據上傳後會進行多重驗證（如：數值範圍檢查、單位一致性），並與證據金庫 (Vault) 連結，確保原始憑證的不可篡改性，提升數據的公信力。

5.  功能項目解說和使用技術 (Features & Tech Stack)
    數據自動化採集與 OCR： 支援 PDF/圖片中的結構化數據提取。技術使用 Google Cloud Vision API 或 AWS Textract 進行文字識別，結合客製化規則引擎進行數據解析與結構化。
    GHG Scope 1-3 計算引擎： 根據 IPCC 準則與在地化排放因子，實時計算溫室氣體排放量。技術使用 Python 後端服務 (如 FastAPI) 搭配 Pandas 進行數據處理與計算，並透過 Celery 處理大量批次計算任務。
    互動式環境儀表板： 提供多維度數據篩選、鑽取與可視化。技術使用 React 搭配 ECharts 或 Highcharts 進行圖表渲染，利用 React Query 進行高效數據緩存與同步。
    數據驗證與異常警示： 自動檢測數據輸入錯誤、異常波動或單位不符。技術使用 Zod 或 Yup 進行前端表單驗證，後端則透過 Apache Flink 或 Kafka Streams 進行實時數據流異常檢測。

6.  品質達標與驗收紅線 (QA Red Lines)
    🚨 UI 跑版紅線： 當數據表格包含超過 500 行數據時，表格的虛擬滾動 (Virtual Scrolling) 必須正常運作，確保頁面流暢度，且表頭與表身對齊不可錯位。
    🚨 邏輯/體驗紅線：
        *   GHG 排放量計算結果與原始數據的單位轉換必須 100% 正確，若因排放因子缺失或數據格式錯誤導致計算失敗，必須明確提示錯誤原因而非顯示為零。
        *   當使用者上傳的數據量過大（例如超過 100MB 的 Excel），系統必須提供進度條與預估等待時間，避免頁面長時間無響應導致使用者誤以為系統崩潰。
        *   任何關鍵數據（如排放因子）的修改，必須觸發相關 GHG 數據的重新計算，並在儀表板上即時反映，確保數據一致性。

7. 矩陣關聯 (Matrix Connection)
上游數據： 來自 `/api-setup` 與 `/data-sources` 的原始能耗數據。
下游影響： 直接提供數據給 `/finance` 計算 ROI 與 `/dashboard` 的環境績效 KPI。
依賴組件： BrandKPIWidget, BrandDataTable, BrandLineChart, BrandFileUploadZone.
