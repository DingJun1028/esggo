# AI 智能平台 AI Platform [AI Platform]
路徑： /ai-platform | 權限： ADMIN, PM, DEVELOPER | 所屬旅程： VI. 系統配置與 AI 治理

1. 模組定位 (Core Purpose)
AI 智能平台是 ESG GO 的核心 AI 治理與配置中心，提供企業級 AI 服務的統一管理介面，確保 AI 模型的合規性、成本效益與高效運行。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 企業的 AI 管理員、永續專案經理 (PM) 或開發者。他們需要管理多個 AI 模型（如生成式 AI、分析模型），確保其在 ESG 報告撰寫、數據分析等場景中穩定、安全、合規地運行，同時有效控制成本，並能追溯 AI 決策的依據。
體驗高光時刻 (Aha Moment)： 當管理員在平台上輕鬆配置完新的 AI 模型參數，並在成本監控儀表板上看到預估成本即時更新，且系統自動提示該配置符合最新的數據隱私規範時，會感到「原來管理複雜的 AI 服務可以如此簡單且安心」。
操作軌跡：
1. AI 管理員登入平台，從左側導航選擇「AI 智能平台」。
2. 進入「模型配置」頁面，選擇要調整的 Gemini 2.0 模型。
3. 透過表單調整模型的溫度 (temperature) 和最大輸出 token 數，並設定每月預算上限。
4. 點擊「保存配置」，系統即時顯示配置生效，並在「成本監控」儀表板上更新預估費用。
5. 切換到「服務狀態」頁面，確認所有 AI 服務均正常運行，且無異常告警。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 採用「左側導航欄 + 右側主內容區」的標準管理後台佈局。主內容區根據功能分為「模型配置卡片區」、「實時監控儀表板」與「日誌查詢表格」。
核心液態玻璃元件： `ModelConfigCard` (帶有可折疊參數區塊的配置卡片)、`CostMonitorGauge` (實時顯示 AI 服務成本與預算達成率的儀表板元件)、`ServiceStatusBadge` (顯示 AI 服務健康狀態的圓點標籤，綠/黃/紅)。
行動端適配 (RWD)： < 768px 時，左側導航欄自動收合為「漢堡選單」。主內容區的「模型配置卡片」會自動堆疊，並將複雜的配置表單轉化為「分步引導式 (Step-by-step Wizard)」流程。實時監控儀表板會自動隱藏次要指標，僅顯示核心成本與服務狀態，並提供「點擊展開詳細圖表」選項。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向：
*   AI 模型配置參數 (如 API Key, 模型版本, 溫度, token 限制) 寫入 `ai_model_configs` 資料表。
*   AI 服務的實時使用量 (token 數、API 調用次數) 從各服務模組匯總，寫入 `ai_usage_logs` 資料表。
*   成本數據根據使用量與預設費率計算，存入 `ai_cost_metrics` 資料表，並影響 `Dashboard KPI` 中的 AI 營運成本指標。
*   AI 服務的運行狀態與錯誤日誌，寫入 `ai_service_health` 資料表。
5T 實踐點：
[T1 Tangible 具體]： 將抽象的 AI 模型參數與服務狀態，具體化為可視化的配置表單、實時儀表板與健康狀態指示燈。例如，將每月 AI 服務的預估成本以具體數字和進度條形式呈現。
[T2 Traceable 追溯]： 任何對 AI 模型配置的修改（如調整參數、更新 API Key），系統都會自動記錄操作者、時間戳與修改前後的值，存入 `ai_config_audit_log`，確保所有變更可追溯。
[T3 Trackable 追蹤]： 實時追蹤各 AI 服務的 API 調用頻率、token 消耗量與錯誤率，並在儀表板上以圖表形式呈現，便於管理員監控 AI 服務的運行效能與成本。
[T4 Transparent 透明]： 平台提供 AI 模型版本管理功能，明確標示當前使用的模型版本及其特性。對於 AI 服務的成本計算邏輯，提供詳細的費率說明與計算公式，確保成本透明可查。
[T5 Trustworthy 信任]： 嚴格的權限管理機制確保只有授權人員才能配置 AI 服務。所有與外部 AI 服務的 API 密鑰都經過加密儲存，並定期輪換。平台會對 AI 模型的輸出進行合規性掃描，防止生成不當或不符合 ESG 標準的內容。

5. 功能項目解說和使用技術 (Features & Tech Stack)
AI 模型集成與配置： 提供統一介面管理多個 AI 模型（如 Gemini 2.0, OpenAI GPT 系列）的 API Key、版本選擇、參數調整（如 temperature, top_p, max_tokens）。
*   技術： 前端使用 `React Hook Form` 搭配 `Zod` 進行表單驗證與狀態管理；後端使用 `OpenAPI Generator` 生成各 AI 服務的 SDK 客戶端，並透過 `Supabase Edge Functions` 處理敏感 API Key 的安全調用。
AI 服務監控與成本分析： 實時顯示各 AI 服務的調用次數、token 消耗、錯誤率及預估成本，並提供歷史數據趨勢圖與預算告警功能。
*   技術： 前端使用 `Recharts` 或 `D3.js` 渲染互動式圖表；後端使用 `Prometheus` 收集 AI 服務的運行指標，並透過 `Grafana` 進行可視化，或直接由 `Supabase Realtime` 訂閱 `ai_usage_logs` 的變更。
OmniBlue 合規服務管理： 允許企業接入 OmniBlue 等第三方合規服務，對 AI 生成內容進行自動審核與風險評估。
*   技術： 使用 `OAuth2` 協議進行安全授權與連接；後端透過 `Webhooks` 接收 OmniBlue 的實時審核結果，並更新相關內容的合規狀態。
權限與安全管理： 實施基於角色的訪問控制 (RBAC)，確保只有具備相應權限的用戶才能查看或修改 AI 配置。所有敏感數據（如 API Key）均加密儲存。
*   技術： 後端使用 `Supabase Row Level Security (RLS)` 實現精細化權限控制；敏感配置數據使用 `PostgreSQL pgcrypto` 擴展進行加密。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線：
*   當 AI 模型配置表單的參數數量過多時，表單元素必須保持對齊，不可出現重疊或溢出容器的情況。
*   實時監控儀表板在數據量激增時，圖表軸標籤與數據點必須清晰可讀，不可出現文字重疊或圖例消失的問題。
🚨 邏輯/體驗紅線：
*   AI 服務配置保存後，必須在 5 秒內在前端顯示配置生效狀態，且後端服務必須在 30 秒內完成配置更新並生效，否則視為配置失敗。
*   當 AI 服務的實際成本達到預設預算上限的 80% 時，系統必須自動觸發郵件或站內通知告警給相關管理員，不可靜默超支。
*   若與外部 AI 服務（如 Gemini）的 API 連接失敗，系統必須在 3 秒內顯示明確的錯誤提示，並提供重試或聯繫客服的選項，不可導致頁面卡死或白屏。
*   任何未經授權的用戶嘗試修改 AI 配置時，必須被系統嚴格攔截，並在後台生成安全事件日誌。
*   AI 模型配置的 API Key 儲存必須經過加密，且在前端介面顯示時必須遮蔽（如顯示 `********`），不可明文顯示。
