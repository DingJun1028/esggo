# 證據金庫 Vault [Vault]
路徑： /vault | 權限： ALL_USERS | 所屬旅程： V. 確信審計與發佈

1. 模組定位 (Core Purpose)
證據金庫是 ESG GO 平台所有佐證文件（如電費單、ISO 證書、採購合約）的加密儲存與查核中心，是支撐平台「可信度」的最底層防線。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 各部門填報窗口與外部稽核員。填報者常找不到最新版憑證，或不確定檔案是否正確；稽核員則總是懷疑「這張截圖有沒有被 Photoshop 修過？」，雙方處於不信任且溝通成本極高的狀態。
體驗高光時刻 (Aha Moment)： 填報者將 PDF 拖入區塊的瞬間，系統伴隨清脆音效亮起綠色的 ZKP Verified 與專屬 Hash 碼。稽核員看到這串由密碼學保證的標記時，立刻放下了戒心，雙方達成零摩擦的信任。
操作軌跡：
1. 填報窗口將供應商提供的 PDF 拖曳至 Dropzone。
2. 系統讀取檔案並展示 SHA-256 雜湊運算的動畫進度條 (增加科技儀式感)。
3. 選擇該文件要關聯的數據項目（例如：一月份台北廠電力）。
4. 檔案入庫，狀態轉為綠色的「已封印 (Sealed)」。
5. 外部稽核員點擊報告中的佐證連結，直接跳轉至金庫中的該文件，並檢視其 Hash 碼與驗證狀態。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 採用「檔案總管型表格 (Data Table)」佈局，結合頂部區域的「全寬拖曳上傳區 (Dropzone)」。表格包含檔名、GRI 條款、檔案大小、上傳者、上傳時間、Hash 值與 ZKP 驗證狀態。
核心液態玻璃元件： VaultOmniTable (萬能聖碑表格，支援篩選、排序與分頁)、BrandStatusDot (顯示 ZKP 驗證狀態：綠=已封印，紅=檔案損毀/失效)、LiquidDropzone (拖曳上傳區，具備動態邊框高亮效果)。
行動端適配 (RWD)： < 768px 時，表格必須自動轉化為「堆疊式資料卡片 (Stacked Cards)」，隱藏複雜的 Hash 字串，僅保留檔名、GRI 條款、檔案大小與 5T 狀態燈號。拖曳上傳區則簡化為「點擊上傳」按鈕。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 使用者在前端瀏覽器上傳檔案時，即刻計算 SHA-256 Hash 值。實體檔案上傳至 Supabase Storage 或 AWS S3 等物件儲存服務。檔案的 Metadata (檔名、大小、上傳者、上傳時間、SHA-256 Hash、ZKP 狀態、關聯 GRI 條款) 寫入 `evidence_vault` 資料表。ZKP 驗證狀態由後端服務定期或觸發式地與區塊鏈或信任第三方服務進行比對，並更新至 `evidence_vault` 資料表。此資料流確保了佐證文件的完整性與可追溯性，並直接影響報告的合規性與可信度。
5T 實踐點：
[T1 Tangible 具體]： 將抽象的「文件信任」具體化為可視的 SHA-256 Hash 值、ZKP 驗證狀態燈號，以及文件與特定 GRI 條款或數據點的明確關聯，讓使用者能直觀理解其可信度。
[T2 Traceable 追溯]： 每個上傳的文件都會記錄其上傳者、上傳時間，並可明確關聯至一個或多個 GRI 條款或數據點。任何對文件的修改嘗試都會被記錄，並導致 Hash 值不匹配，確保來源可追溯。
[T3 Trackable 追蹤]： 系統會自動記錄每個文件的生命週期事件，包括上傳、關聯、ZKP 驗證結果更新，以及任何嘗試修改或刪除的行為，形成不可篡改的稽核日誌 (Audit Log)，供後續審計追蹤。
[T4 Transparent 透明]： 文件的 SHA-256 Hash 值與 ZKP 驗證狀態在介面上清晰可見，供所有具備權限的使用者（包括外部稽核員）查閱，確保資訊公開透明，無隱藏資訊。
[T5 Trustworthy 信任]： 檔案拖入前端瀏覽器的瞬間，即刻計算出 SHA-256 Hash，並隨檔案上傳至資料庫。後續對檔案的任何微小修改，都會導致 Hash 對不上而亮起紅燈警報，從而保證文件的完整性與不可篡改性。ZKP 驗證進一步提供零知識證明，強化信任機制。

5. 功能項目解說和使用技術 (Features & Tech Stack)
客戶端 Hash 運算： 為了減少伺服器負擔與傳輸竄改風險，在前端直接計算檔案的 SHA-256 Hash。技術使用瀏覽器原生的 Web Crypto API (`crypto.subtle.digest`)。
拖曳上傳互動與預覽： 支援多檔案拖曳上傳、進度條顯示與上傳前預覽。技術使用 `react-dropzone` 搭配客製化 Liquid Glass 樣式邊框高亮，並結合 `FileReader` 進行本地預覽。
大檔案串流上傳： 避免超過 50MB 的 PDF 或影片檔案卡死瀏覽器或造成伺服器超時。技術使用 Supabase Storage 或 AWS S3 的分塊上傳 (Multipart Upload) 或 TUS (Resumable Uploads) 協議。
ZKP 驗證狀態顯示： 前端根據後端提供的 ZKP 驗證結果，動態顯示綠色「已封印」或紅色「驗證失敗」狀態。後端服務可能整合區塊鏈或信任時間戳服務進行驗證，並透過 WebSocket 或 Server-Sent Events (SSE) 即時更新前端狀態。
文件關聯與篩選： 允許使用者將文件關聯至特定的 GRI 條款、數據點或專案。前端表格支援多維度篩選與關鍵字搜尋。技術使用 React Query 進行資料快取與優化，搭配 `react-table` 或客製化表格組件，並利用後端全文檢索 (Full-Text Search) 引擎（如 PostgreSQL 的 `tsvector` 或 ElasticSearch）提升搜尋效率。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線：
*   檔名或 Hash 字串過長時必須觸發 `text-overflow: ellipsis` (顯示...)，配合 Tooltip 檢視完整內容，絕對不可撐開表格導致整個頁面出現醜陋的 X 軸滾動條。
*   在行動端 (RWD) 模式下，堆疊式資料卡片必須確保所有關鍵資訊（檔名、狀態、GRI 條款）清晰可見，且卡片間距適中，避免擠壓或重疊。
*   拖曳上傳區 (Dropzone) 在檔案拖入時，其邊框高亮效果必須流暢且無閃爍，且在檔案離開時能正確恢復原狀。
🚨 邏輯體驗紅線：
*   若系統檢測到使用者上傳了重複的檔案 (Hash 完全相同)，必須阻斷上傳並彈出友善提示：「此文件已存在於金庫中，是否直接建立關聯？」，以節省儲存空間與防止資料冗餘。
*   大檔案上傳過程中，若發生網路中斷，系統必須能夠自動暫停並在網路恢復後「續傳」，而非直接失敗並要求使用者重新上傳。同時，應提供清晰的進度指示。
*   任何嘗試修改已「封印 (Sealed)」文件的行為，無論是透過前端介面或直接 API 呼叫，都必須被後端嚴格拒絕，並返回明確的錯誤訊息（例如：HTTP 403 Forbidden），不可靜默失敗或允許修改。
*   對於上傳失敗的檔案，必須提供清晰的錯誤原因（例如：檔案格式不支援、檔案過大、網路錯誤、伺服器儲存空間不足），而非籠統的「上傳失敗」，並引導使用者解決問題。
*   ZKP 驗證失敗時，系統必須立即將文件標記為「驗證失敗」，並提供詳細的失敗原因，同時觸發管理員警報。
