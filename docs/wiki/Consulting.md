# ESG 顧問服務模組 ESG Consulting Hub [Consulting]
路徑： /consulting | 權限： ADMIN, PM, CSO | 所屬旅程： VI. 外部協作與服務

1. 模組定位 (Core Purpose)
ESG Consulting Hub 旨在為企業提供一站式 ESG 顧問服務媒合與管理平台，協助企業快速連結專業顧問，獲得客製化的永續策略規劃、報告撰寫輔導與合規審查支援。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 企業永續部門主管 (PM, CSO)。他們在 ESG 轉型初期，常因缺乏內部專業知識或人力，需要外部顧問協助，但尋找可靠顧問耗時費力，服務內容不透明，溝通效率低，且難以追蹤服務進度與成果。
體驗高光時刻 (Aha Moment)： 當 PM 提交服務需求後，系統立即推薦 3 位預先審核的專業顧問，並提供透明的報價與服務時程。PM 瞬間感受到「終於有清晰的解決方案了！」的安心感。
操作軌跡：
1. 企業用戶從左側導航點擊「顧問服務」，瀏覽不同類型的 ESG 顧問服務（如：GRI 報告輔導、碳盤查）。
2. 選擇感興趣的服務，填寫客製化需求表單，系統自動推薦 3 位符合條件的顧問。
3. 比較顧問資歷、報價與服務方案，選擇一位發送合作邀請。
4. 在平台內與顧問進行初步溝通，確認服務範圍與里程碑。
5. 簽署電子合約，並在平台追蹤服務進度與交付成果。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 採用「服務卡片網格 (Service Card Grid)」展示顧問服務，點擊卡片進入「服務詳情與需求表單」頁面，右側可能會有「推薦顧問列表」或「服務進度儀表板」。
核心液態玻璃元件： BrandServiceCard (展示服務類型、價格區間、預估時長)、ConsultantProfileBadge (顧問頭像、專長、評分)、DynamicFormBuilder (用於客製化需求表單)。
行動端適配 (RWD)： < 768px 時，服務卡片網格自動轉為「單欄堆疊」，需求表單的複雜輸入框（如日期選擇器、多行文本）應優化為行動友善的元件，確保易於填寫。服務進度儀表板應轉化為「列表式進度條」。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 用戶提交的服務需求 (service_requests) 寫入資料庫，觸發顧問媒合服務。顧問的服務方案、資歷 (consultant_profiles) 儲存於資料庫。服務進度與交付物 (consulting_projects, project_deliverables) 儲存於資料庫，並與用戶的 ESG 報告草稿 (report_drafts) 進行關聯。
5T 實踐點：
[T4 Transparent 透明]： 顧問的資歷、過往案例、服務報價與用戶評價，都應在平台上公開透明展示，供用戶比較選擇。所有服務條款與費用結構清晰可見。
[T3 Trackable 追蹤]： 每個顧問服務項目都應有明確的里程碑與交付物清單，用戶可在平台追蹤服務進度，並查看顧問上傳的階段性成果。所有溝通紀錄與文件交換都應被記錄在案，可供日後審查。
[T5 Trustworthy 信任]： 平台對所有入駐顧問進行嚴格的資質審核與背景調查，確保其專業性與合規性。服務完成後，用戶可對顧問進行匿名評價，確保服務品質的可靠性與持續改進。

5. 功能項目解說和使用技術 (Features & Tech Stack)
顧問服務媒合引擎： 根據用戶需求（產業、服務類型、預算）與顧問專長、可用性進行智能匹配。技術使用 ElasticSearch 進行顧問資料的快速檢索與排序，結合簡單的加權匹配演算法。
線上溝通與文件協作： 提供平台內即時聊天、視訊會議排程與文件共享功能。技術使用 WebSocket (如 Socket.IO) 實現即時通訊，整合 WebRTC 進行視訊，並利用 Supabase Storage 或 S3 進行文件安全共享與版本控制。
電子合約簽署與進度管理： 支援線上簽署服務合約，並提供甘特圖或看板視圖追蹤服務進度。技術使用 DocuSign API 或自建基於 Web Crypto API 的電子簽名功能，前端使用 React Flow 或類似庫展示進度。
服務評價與反饋機制： 允許用戶對已完成的顧問服務進行評分與文字評價。技術使用 Star Rating UI 元件，後端將評價數據寫入 consultant_reviews 表，並計算顧問的平均得分，同時提供文字評論的審核機制。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線： 顧問服務卡片在不同螢幕尺寸下，圖片與文字比例必須保持一致，不可出現圖片拉伸變形或文字溢出卡片邊界。需求表單中的必填欄位若未填寫，提交時必須有清晰的紅色錯誤提示，且提示文字不可被其他 UI 元素遮擋。
🚨 邏輯/體驗紅線： 顧問媒合結果必須在 5 秒內呈現，若無匹配顧問，應提供清晰的「無可用顧問」提示，並引導用戶調整需求或聯繫客服，不可出現空白或錯誤頁面。電子合約簽署流程必須確保法律效力，任何簽署失敗或篡改嘗試都應被記錄並發出警報，且用戶必須收到明確的失敗原因提示，而非靜默失敗。服務進度更新必須即時反映在用戶介面上，不可出現顧問已更新進度但用戶端仍顯示舊狀態的數據不同步問題。
