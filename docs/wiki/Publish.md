# SustainPublish 報告發佈中心 SustainPublish [Publish]
路徑： /publish | 權限： PM, ADMIN, CSO | 所屬旅程： V. 確信審計與發佈

1. 模組定位 (Core Purpose)
SustainPublish 是 ESG GO 的最終報告發佈中樞，負責將編輯完成的永續報告，透過 ZKP (零知識證明) 確保其內容完整性與可信度，並以多種格式（如 PDF、互動式網頁）安全地發佈。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 永續專案經理 (PM)、永續長 (CSO) 及高階主管。他們需要將耗時數月撰寫的永續報告，以專業、合規且具備公信力的方式對外發佈。痛點在於確保最終報告的內容與佐證資料一致，避免發佈後被質疑數據真實性，同時還要兼顧不同利害關係人對報告格式的需求（例如：董事會需要精美 PDF，投資人需要可互動的線上版本）。
體驗高光時刻 (Aha Moment)： 當 CSO 點擊「發佈」按鈕後，系統在 5 秒內自動生成一份帶有區塊鏈時間戳與 ZKP 驗證碼的精美 PDF 報告，同時自動更新企業官網上的 ESG 報告專區。CSO 看到報告的完整性與發佈效率得到極大提升時，感受到「這份報告的公信力，現在由科技來保證」的巨大安心感。
操作軌跡：
1. 從左側導航選擇「報告發佈中心」。
2. 選擇欲發佈的報告版本（例如：2023 年度最終版）。
3. 檢閱系統自動生成的「發佈前合規檢查清單」，確認所有項目皆為綠燈。
4. 選擇發佈格式（PDF、Web）與發佈範圍（內部預覽、公開發佈）。
5. 點擊「啟動 ZKP 驗證與發佈」，等待系統完成處理。
6. 收到發佈成功通知，並可下載最終報告或取得線上報告連結。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 採用「左側報告預覽區 + 右側發佈設定面板」的雙欄佈局。預覽區佔 60% 寬度，提供報告的即時渲染效果；設定面板佔 40% 寬度，包含版本選擇、格式選項、ZKP 狀態與發佈按鈕。
核心液態玻璃元件： ReportPreviewer (內嵌 PDF.js 或客製化 HTML 渲染器)、PublicationOptionsCard (包含下拉選單、開關按鈕)、ZKPStatusBadge (顯示驗證進度與結果)。
行動端適配 (RWD)： < 768px 時，右側「發佈設定面板」自動收合為底部懸浮的 FAB (Floating Action Button)，點擊後以全螢幕 Modal 彈出。左側「報告預覽區」則自動調整為 100% 寬度，並提供手勢縮放與翻頁功能。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 從 `report_drafts` 資料表讀取最終報告內容，從 `evidence_vault` 讀取相關佐證文件 Hash。生成最終報告後，將其儲存至 Supabase Storage / S3，並將發佈紀錄 (包含報告 Hash, 發佈時間, 發佈者, ZKP 證明連結) 寫入 `published_reports` 資料表。
5T 實踐點：
[T5 Trustworthy 信任]： 報告發佈前，系統會對報告的最終內容進行 SHA-256 雜湊運算，並將此 Hash 值與時間戳一同上鏈（或整合至信任時間戳服務），生成 ZKP 證明。任何發佈後的內容竄改，都會導致 Hash 值不符，從而使 ZKP 驗證失敗，確保報告的不可篡改性與公信力。
[T3 Trackable 追蹤]： 每次報告發佈行為，都會在 `published_reports` 資料表中留下詳細紀錄，包含發佈者、發佈時間、報告版本、發佈格式、ZKP 證明 ID 等，確保所有發佈動作皆可追溯。
[T4 Transparent 透明]： 報告發佈後，系統會提供一個公開的 ZKP 驗證連結，任何利害關係人都可以透過該連結，獨立驗證已發佈報告的內容是否與原始證明一致，提升報告的透明度。

5. 功能項目解說和使用技術 (Features & Tech Stack)
報告版本管理與選擇： 允許使用者選擇歷史草稿或已鎖定版本進行發佈。技術使用 Supabase 的版本控制功能或自定義的 `report_versions` 資料表，前端使用 React Select 或 Ant Design Select 元件。
多格式報告生成：
*   **PDF 報告生成：** 伺服器端渲染 (SSR)。技術使用 Puppeteer 或 Headless Chrome 搭配 Node.js，將最終 HTML 內容轉換為高解析度 PDF，確保排版與字體一致性。
*   **互動式網頁報告：** 透過 Next.js 或 Astro 的靜態網站生成 (SSG) 功能，將報告內容預渲染為優化的 HTML 頁面，提供快速載入與 SEO 友善的線上體驗。
ZKP 報告完整性驗證：
*   **內容雜湊運算：** 伺服器端使用 Node.js 的 `crypto` 模組計算報告內容的 SHA-256 Hash。
*   **區塊鏈/時間戳整合：** 透過 Web3.js 或專用 API (例如：Factom, OpenTimestamps) 將報告 Hash 上鏈或進行時間戳服務整合，生成不可篡改的證明。
發佈前合規檢查： 系統自動掃描報告內容，比對預設的 GRI/ISSB 指標完成度、數據引用完整性等。技術使用客製化規則引擎 (Rule Engine) 搭配後端數據分析服務。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線： PDF 預覽器在不同解析度下，報告內容的文字、圖片與表格必須保持正確的換行與對齊，絕對不可出現文字溢出容器、圖片變形或表格跨頁斷裂導致閱讀困難的問題。
🚨 邏輯/體驗紅線：
*   ZKP 驗證失敗時，系統必須清晰地告知使用者失敗原因（例如：內容被竄改、證明鏈斷裂），並提供重新驗證或聯繫客服的明確指引，而非僅顯示錯誤代碼。
*   報告發佈流程中，若因網路中斷或伺服器錯誤導致發佈失敗，必須提供斷點續傳機制或友善的錯誤提示，並確保已生成的 ZKP 證明不會因此失效，避免使用者重複操作或產生不信任感。
*   已發佈的報告，其內容在系統內部必須被鎖定為 Read-Only 狀態，任何嘗試修改已發佈報告內容的行為都應被系統拒絕並給予明確提示。
