ESG GO 平台核心原則與技術基石 ESG GO Platform Core Principles
路徑： /system-overview | 權限： ALL_USERS | 所屬旅程： I. 系統概覽與導入

1. 模組定位 (Core Purpose)
本頁面旨在闡述 ESG GO 平台作為企業級 SaaS 解決方案的核心原則與技術基石，確保系統在功能完整性、架構一致性、數據可信度與使用者體驗上，達到業界領先的「Production-Ready」標準，為所有模組提供穩定、可靠且可持續擴展的基礎。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 企業決策者、永續專案經理、稽核員及所有 ESG GO 系統使用者。他們需要一個高度可靠、易於使用且數據可信的平台來管理其 ESG 報告與治理流程，避免因系統不穩定、數據不一致或視覺混亂導致的信任危機與效率低下。
體驗高光時刻 (Aha Moment)： 當使用者在不同裝置、不同模組間切換時，發現無論是視覺風格、操作邏輯還是數據呈現，都保持著高度的一致性與流暢性，並且確信所有數據都經過嚴格的 5T 協定驗證，從而感受到「這是一個值得信賴且專業的平台，我的 ESG 治理任務可以安心交付」。
操作軌跡：
1.  新使用者首次登入平台，感受到統一且精緻的視覺風格，快速理解操作模式。
2.  在不同裝置（手機、平板、桌面）上操作，發現介面始終保持優雅且功能完整，無須重新適應。
3.  檢視關鍵數據報告時，能清晰看到數據來源與追溯鏈，對其真實性深具信心。
4.  在任何模組中，都能體驗到流暢的數據同步與一致的互動模式，提升工作效率。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 採用「左側導航欄 + 右側主內容區」的經典佈局，提供清晰的資訊層級與導航效率。主內容區內部則廣泛應用「Bento Grid」組織方式，確保資訊模組化、易於掃描且視覺平衡。
核心液態玻璃元件：
*   `ClientLayout`: 全局佈局容器，負責響應式切換與 Safe Area 優化。
*   `LiquidGlassCard`: 應用於所有主要內容區塊，提供統一的視覺深度與質感。
*   `BrandButton`: 統一風格的互動按鈕，確保點擊區域與視覺回饋一致。
*   `BrandInput`: 統一風格的輸入框，提供清晰的狀態提示（如聚焦、錯誤）。
*   `BrandNav`: 桌面側邊欄與行動端底部導航，確保導航邏輯與視覺一致。
*   **品牌色配置:**
    *   主色：`#003262` (Berkeley Blue) - 應用於主要導航、標題與核心互動元素，傳達專業與穩重。
    *   輔色：`#FDB515` (California Gold) - 應用於強調元素、進度條與高光提示，增加活力與引導性。
*   **視覺標準:** 嚴格遵循「Berkeley Academic Precision」原則，確保介面簡潔、專業、資訊密度適中，避免視覺噪音，提升資訊傳達效率。
行動端適配 (RWD)：
*   **全局適配:** 所有頁面均透過 `ClientLayout` 實現響應式設計，確保在不同螢幕尺寸下都能提供最佳體驗。
*   **導航切換:** 當螢幕寬度 < 768px 時，桌面端側邊欄自動收合，並切換為底部導航列，提供核心功能快速入口，避免佔用寶貴的螢幕空間。
*   **內容重排:** 頁面內容（如複雜表格、多欄佈局）會自動重排為單欄堆疊或卡片式佈局，確保在小螢幕上文字可讀性與操作性，防止橫向滾動。
*   **Safe Area 優化:** 針對 iOS/Android 裝置的瀏海屏、底部手勢區進行優化，確保內容不被遮擋，提供無縫的視覺體驗。
*   **全域 RWD 響應式版型:** 確保在任何裝置上，系統都能提供最佳的視覺與操作體驗，防止文字溢出、按鈕錯位等常見跑版問題。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 平台所有數據操作，無論是前端輸入、AI 生成、還是後端批次處理，均統一透過 `lib/db.ts` 作為核心數據接口層，與 Supabase 進行交互。這確保了數據從輸入到儲存、再到展示的整個生命週期中，都遵循一致的業務邏輯與資料模型。所有數據變更都會觸發相關 Dashboard KPI 的即時或近即時更新，確保管理層能隨時掌握最新動態。
5T 實踐點： ESG GO 系統將 5T 協定深度整合至所有涉及數據操作的模組中，確保平台數據的最高可信度與治理標準。
*   **[T1 Tangible 具體]**: 平台將抽象的 ESG 指標與數據，透過標準化的輸入介面、可視化的圖表（如重大性矩陣、排放儀表板）和具體的佐證文件連結，轉化為可量化、可感知的具體成果。例如，碳排放數據不僅是數字，還能連結到具體的電費單與計算方法，讓數據不再是空泛的數字。
*   **[T2 Traceable 追溯]**: 系統為所有關鍵數據與操作提供完整的追溯鏈。每一次數據的修改、文件的上傳、AI 的生成決策，都會在後台留下詳細的 Audit Log，記錄操作者、時間戳、變更內容與相關上下文，確保數據來源與演變過程清晰可查，滿足稽核要求。
*   **[T3 Trackable 追蹤]**: 平台提供全面的進度追蹤與任務管理功能。永續報告的撰寫進度、各部門的數據填報狀態、稽核文件的審核流程，都能被即時監控與追蹤，確保專案按時推進，並能識別潛在瓶頸，提升專案管理效率。
*   **[T4 Transparent 透明]**: AI 賦能的模組（如 SustainWrite 編輯器）會明確標示其生成內容的數據來源與推理依據（例如：引用內部知識庫、外部法規或特定數據點），避免「黑箱作業」。所有數據計算邏輯與模型參數也盡可能公開透明，供稽核方檢閱，建立信任。
*   **[T5 Trustworthy 信任]**: 透過加密技術（如文件上傳時的 SHA-256 Hash 驗證）、嚴格的權限管理（Row Level Security）與數據一致性檢查，確保儲存在證據金庫中的文件與平台上的數據未經篡改，且來源可靠。任何數據異常或篡改嘗試都會被即時標記與警示，建立堅不可摧的信任基礎。

5. 功能項目解說和使用技術 (Features & Tech Stack)
全局數據接口層 (Global Data Interface Layer):
*   **行為:** 統一管理與 Supabase 後端的數據交互，包括 CRUD 操作、實時訂閱與權限控制，確保全系統數據同步與欄位口徑一致。
*   **技術:** `lib/db.ts` 封裝 Supabase JS Client，結合 React Query 進行數據緩存與狀態管理，優化 API 請求與響應速度。
響應式佈局引擎 (Responsive Layout Engine):
*   **行為:** 根據裝置螢幕尺寸動態調整頁面佈局與導航模式，提供一致且優化的跨裝置體驗。
*   **技術:** `ClientLayout` 元件基於 Tailwind CSS 的響應式斷點與 CSS Grid/Flexbox 實現，結合 `useMediaQuery` Hook 進行客戶端尺寸判斷，確保精準適配。
品牌視覺語言實作 (Brand Visual Language Implementation):
*   **行為:** 確保所有 UI 元件嚴格遵循「Liquid Glass」設計哲學與品牌色規範，提供統一且高品質的視覺體驗。
*   **技術:** 使用 Styled Components 或 Emotion 進行原子級 UI 元件的樣式定義，並透過 Storybook 進行設計系統的維護與一致性驗證，加速開發並減少視覺偏差。
5T 協定嵌入框架 (5T Protocol Embedding Framework):
*   **行為:** 在數據輸入、處理與展示的關鍵環節自動植入 5T 相關的元數據與邏輯，確保數據的完整性與可信度。
*   **技術:** 透過 Supabase Trigger Functions 實現數據變更的 Audit Log 記錄 (T2, T3)，前端利用 Context API 或 Zustand 傳遞數據來源標籤 (T4)，並結合 Web Crypto API 進行文件 Hash 驗證 (T5)，形成全棧式的 5T 實踐。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線：
*   在任何主流瀏覽器（Chrome, Firefox, Safari, Edge）的桌面版與行動版（iOS, Android）上，頁面佈局必須保持穩定，不可出現文字溢出、元件重疊、按鈕被遮擋或不必要的 X 軸滾動條。
*   `ClientLayout` 在不同斷點切換時，導航欄與主內容區的過渡動畫必須平滑，不可出現閃爍或內容跳動。
*   所有 `LiquidGlassCard` 容器必須確保內部內容在溢出時能正確觸發 Y 軸滾動條，而非撐破容器，保持視覺完整性。
🚨 邏輯/體驗紅線：
*   **數據一致性破壞:** 任何模組間的數據同步延遲超過 500ms，或因數據源不一致導致的顯示錯誤，必須亮紅燈。這包括 Dashboard KPI 未能即時反映最新數據。
*   **5T 協定失效:**
    *   若 Audit Log 未能正確記錄關鍵數據變更 (T2, T3)。
    *   AI 生成內容未能標示來源或標示錯誤 (T4)。
    *   文件 Hash 驗證失敗但系統仍標示為「已信任」 (T5)。
    *   抽象數據未能連結到具體實證或連結錯誤 (T1)。
    *   以上任何情況都視為嚴重邏輯錯誤，必須立即修復。
*   **斷網處理:** 在斷網情況下，系統必須提供友善的離線提示，並盡可能支援離線操作（如草稿暫存至 IndexedDB），不可出現空白頁面或未處理的 Error 500 白畫面，避免使用者恐慌。
*   **權限錯誤:** 未經授權的角色成功訪問或修改了受限內容，或授權角色無法訪問其應有權限的內容，這將構成嚴重的安全漏洞。
