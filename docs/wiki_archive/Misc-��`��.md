SustainWrite 編輯器 SustainWrite Editor
路徑： /editor | 權限： PM, ADMIN | 所屬旅程： IV. AI 賦能與撰寫

1. 模組定位 (Core Purpose)
SustainWrite 是 ESG GO 的核心撰寫中樞，提供 208 頁報告框架、佐證清單與 AI 合規協作，幫助企業高效率產出符合 GRI/ISSB 標準的可信永續報告。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 永續專案經理 (PM)。他們通常對著 200 多頁的 Word 檔案發愁，需要不斷在各種 PDF 憑證與法規網頁間頻繁切換，深怕寫錯或漏掉 GRI 指標。
體驗高光時刻 (Aha Moment)： 當 PM 點擊「AI 擴寫」時，系統不僅在 3 秒內生成合規草稿，還自動在右側抽屜亮起該段落對應的「電費單 PDF 縮圖」。PM 瞬間感受到「不用自己大海撈針」的巨大釋放感。
操作軌跡：
1. 焦慮地從左側導航樹點開尚未完成的 GRI 305 章節。
2. 召喚 AI 助手，請其依據今年度數據生成排放聲明。
3. 檢閱 AI 內容與標註的證據來源，確認無誤後點擊「寫入」。
4. 執行合規掃描，看到全綠燈通過，安心存檔。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 採用「左中右三欄佈局」。左側為「章節導航」，中央為「所見即所得編輯區」，右側為「AI 顧問 / 佐證文件抽屜」。
核心液態玻璃元件： 使用 BrandFloatingAgent 作為右下角懸浮 AI 助手；編輯區背景採用 glassBg (白/深藍磨砂)。
行動端適配 (RWD)： < 768px 時，左右兩欄自動隱藏為「漢堡選單」與「懸浮 FAB」，中央編輯區必須 100% 滿版，防止文字擠壓破版。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 編輯內容自動每 30 秒 Autosave 至 Supabase report_drafts 表。
5T 實踐點：
[T4 Transparent 透明]： AI 生成的每一段文字，旁邊都會生成一個 BrandBadge，標示其推理來源（內部知識庫或外部法規）。
[T2 Traceable 追溯]： 使用者在文字中 @ 標記數據時，自動建立與 evidence_vault 的雙向連結。

5. 功能項目解說和使用技術 (Features & Tech Stack)
所見即所得編輯 (WYSIWYG)： 支援標題、清單、表格與 @ 標記。技術使用 TipTap 或 Lexical 編輯器核心進行客製化。
AI 擴寫與合規掃描： 透過背景串接 AI 進行內容分析。技術使用 Genkit 搭配 Gemini 2.0 Pro 模型，並以 Server-Sent Events (SSE) 實作打字機串流效果。
自動儲存防呆機制： 背景靜默儲存。技術使用 Supabase Realtime 與 Lodash debounce 控制 API 請求頻率。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線： 當 AI 一次性生成超過 2000 字時，中央編輯區的 Y 軸滾動條必須正常出現，絕對不可撐破外層 Liquid Glass 卡片容器。
🚨 邏輯/體驗紅線： AI 回傳結果若包含違背 ESG 標準的幻覺字眼，必須被攔截並標示為紅字。若發生斷網，必須平滑切換至「離線暫存模式 (IndexedDB)」，不可彈出令使用者恐慌的 Error 500 白畫面。

1. 模組定位 (Core Purpose)
控制台是 ESG GO 系統的個人化首頁，提供使用者一目瞭然的 ESG 績效總覽、關鍵任務提醒與快速操作入口，確保用戶能高效掌握其負責的永續進度。

目標對象與痛點： 所有系統使用者，尤其是永續專案經理 (PM)。他們每天登入系統，需要快速了解當前最重要的任務、數據異常或報告進度，避免遺漏關鍵資訊。
體驗高光時刻 (Aha Moment)： PM 登入後，一眼看到「本月碳排放量較上月下降 5%」的綠色趨勢卡片，同時「待審核文件」的紅色數字提醒也清晰可見，立刻感受到「所有重要資訊都已為我準備好」的安心感。
操作軌跡：
1. 登入系統，自動導向控制台。
2. 快速瀏覽各項 KPI 儀表板，檢查是否有異常數據。
3. 點擊「待辦任務」卡片，直接跳轉至相關功能頁面處理。
4. 檢視最新系統通知與法規更新。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 採用「多卡片網格佈局 (Multi-Card Grid Layout)」，可自訂卡片大小與位置，頂部為全局通知與快速搜尋欄。
核心液態玻璃元件： BrandKPIcard (具備趨勢箭頭與顏色指示)、BrandTaskWidget (可展開收合的任務列表)、BrandNotificationBell (帶有未讀計數)。
行動端適配 (RWD)： < 768px 時，所有卡片自動堆疊為單欄佈局，並可透過左右滑動切換不同類型的資訊流，確保資訊不被截斷。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 各模組的關鍵數據（如 emissions, tasks, report_status）匯總至此，不直接寫入資料，僅作讀取與展示。
5T 實踐點：
[T5 Timely 即時]： 所有 KPI 數據與任務狀態均為實時更新，確保使用者看到的資訊是最新的。
[T1 Tangible 具體]： 將複雜的 ESG 數據轉化為直觀的圖表、進度條和顏色指示，讓抽象的績效變得具體可感。

5. 功能項目解說和使用技術 (Features & Tech Stack)
可自訂儀表板： 允許使用者拖曳、調整卡片大小與位置。技術使用 React Grid Layout 或類似的拖曳庫。
實時數據更新： 確保 KPI 數據能即時反映最新狀態。技術使用 WebSocket 或 Supabase Realtime 訂閱數據變更。
智能任務推薦： 根據用戶角色、權限與數據狀態，推薦相關任務。技術使用 Next.js API Routes 處理後端邏輯，結合 PostgreSQL 查詢。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線： 當用戶自訂卡片佈局後，無論螢幕大小如何變化，卡片之間必須保持一致的間距，不可出現重疊或邊緣溢出。
🚨 邏輯/體驗紅線： 任何 KPI 卡片在數據載入失敗時，必須顯示友善的「數據載入失敗，請稍後再試」提示，而非空白或錯誤碼，並提供刷新按鈕。

數位分身 Digital Twin
路徑： /digital-twin | 權限： PM, ADMIN, CSO | 所屬旅程： I. 核心指揮中心

1. 模組定位 (Core Purpose)
數位分身提供企業 ESG 績效的虛擬模擬環境，透過整合多維度數據，實現情境分析、風險預測與策略優化，幫助企業在決策前預見永續轉型的潛在影響。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 永續長 (CSO) 或高階決策者。他們需要評估不同減碳方案、供應鏈調整或投資策略對 ESG 表現的長期影響，但傳統方法難以量化且耗時。
體驗高光時刻 (Aha Moment)： CSO 在模擬器中將「能源結構」從燃煤調整為太陽能，系統立刻在 3D 廠區模型上顯示碳排放量預計下降 30%，同時預估財務成本與回收期，讓 CSO 驚嘆於「未來已在眼前」的洞察力。
操作軌跡：
1. 選擇一個現有的 ESG 績效模型作為基準。
2. 調整模擬參數，例如：碳排放目標、供應鏈夥伴、投資項目。
3. 運行模擬，觀察各項 ESG 指標的預測變化。
4. 比較不同情境的結果，生成決策報告。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 採用「左側參數控制面板 + 右側 3D 互動式視覺化模型」的雙欄佈局。
核心液態玻璃元件： BrandScenarioSlider (帶有預測範圍顯示的滑桿)、Brand3DModelViewer (支援縮放、旋轉與熱點互動)、BrandImpactGauge (顯示多維度影響)。
行動端適配 (RWD)： < 768px 時，3D 模型預設為靜態截圖，點擊後可進入全螢幕互動模式；參數控制面板則收合為底部 Tab 或可滑動的抽屜。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 讀取 historical_esg_data, asset_inventory, supply_chain_partners 等資料表，透過模擬引擎計算預測結果，不直接寫入生產資料，但可儲存模擬情境至 scenario_library。
5T 實踐點：
[T1 Tangible 具體]： 將抽象的策略影響，透過 3D 視覺化與量化指標，轉化為具體可見的成果預期。
[T3 Trackable 追蹤]： 每次模擬的情境參數、運行時間與結果都會被記錄，方便回溯與比較。

5. 功能項目解說和使用技術 (Features & Tech Stack)
3D 互動模型渲染： 支援複雜的場景渲染與數據疊加。技術使用 Three.js 或 Babylon.js 搭配 React-three-fiber。
情境模擬引擎： 執行多變量分析與預測。技術使用 Python 後端（如 FastAPI）搭配科學計算庫（如 NumPy, SciPy, Pandas）進行模型運算。
數據可視化與比較： 呈現不同情境下的 KPI 變化。技術使用 D3.js 或 ECharts 進行動態圖表渲染。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線： 3D 模型在不同解析度下必須保持正確的長寬比，不可出現模型變形或紋理模糊。
🚨 邏輯/體驗紅線： 當模擬參數輸入超出合理範圍時（例如：碳排放量設為負值），系統必須即時給予輸入驗證錯誤提示，並阻止模擬運行，避免產生無效結果。

企業健檢 Health Check
路徑： /health-check | 權限： PM, ADMIN, CSO | 所屬旅程： I. 核心指揮中心

1. 模組定位 (Core Purpose)
企業健檢提供自動化的 ESG 合規性與績效評估，透過預設的標準框架（如 GRI、ISSB、TCFD），快速識別企業在永續發展上的優勢與潛在風險點，並提供改進建議。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 永續專案經理 (PM)。他們需要定期評估企業的 ESG 表現，但手動比對法規與內部數據耗時費力，且容易遺漏細節。
體驗高光時刻 (Aha Moment)： PM 點擊「執行健檢」後，系統在數秒內生成一份「紅綠燈」報告，清晰標示出「GRI 305-1 數據缺失」的紅色警示，並附上「建議：請上傳最新電費單至證據金庫」的行動建議，讓 PM 感到「問題在哪、怎麼解決，一目了然」。
操作軌跡：
1. 選擇要執行的健檢標準（例如：GRI 2021）。
2. 點擊「開始健檢」，系統自動分析已輸入的數據與文件。
3. 檢視健檢報告，查看合規分數、風險點與改進建議。
4. 點擊建議，直接跳轉到相關數據輸入或文件上傳頁面。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 採用「左側導航樹 (標準章節) + 右側內容區 (健檢結果與建議)」的雙欄佈局。結果區以「進度環 (Progress Ring)」與「紅綠燈狀態標籤 (Status Tags)」為主。
核心液態玻璃元件： BrandProgressRing (顯示整體合規度)、BrandIssueCard (標示問題與建議)、BrandComplianceMatrix (顯示各指標達成狀況)。
行動端適配 (RWD)： < 768px 時，左側導航樹收合為頂部下拉選單，右側內容區則自動堆疊，確保每個問題卡片完整顯示。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 讀取 esg_data, evidence_vault, report_drafts 等資料表，與 compliance_standards 進行比對，生成健檢報告，不寫入核心數據。
5T 實踐點：
[T4 Transparent 透明]： 每個健檢結果都明確標示其判斷依據（哪個標準、哪個數據點），讓使用者清楚了解問題來源。
[T2 Traceable 追溯]： 每次健檢的執行時間、所選標準與結果都會被記錄，方便追溯歷史表現。

5. 功能項目解說和使用技術 (Features & Tech Stack)
自動合規性比對： 根據預設規則引擎，比對數據與標準。技術使用後端服務（如 Node.js 或 Python）搭配規則引擎庫（如 json-rules-engine）。
智能建議生成： 根據健檢結果，提供具體可行的改進建議。技術使用 AI 模型（如 GPT-3.5）進行文本生成，結合預設的知識庫。
互動式報告展示： 允許用戶鑽取查看詳細問題。技術使用 React Context API 進行狀態管理，結合 Recharts 或 Chart.js 渲染圖表。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線： 當建議文字過長時，必須自動換行並保持卡片高度一致，不可導致卡片高度不一或文字溢出。
🚨 邏輯/體驗紅線： 若健檢過程中因數據量過大導致超時，必須顯示「健檢進行中，請稍後查看結果」的非阻斷式提示，並在後台完成後透過通知中心告知用戶，而非讓用戶長時間等待。

體驗高光時刻 (Aha Moment)： PM 在平台上瀏覽專家簡介，發現一位專精於「範疇三排放計算」的專家，立刻預約了 30 分鐘的視訊會議。會議中，專家精準地解答了 PM 的所有疑問，並提供了實用工具，讓 PM 感到「專業支援觸手可及」的效率與安心。

2. 點擊心儀專家，查看詳細簡介與可預約時段。
3. 選擇時段並填寫諮詢需求，完成預約。
4. 在預約時間透過內建視訊會議工具與專家進行諮詢。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 讀取 expert_profiles, booking_schedules 資料表，預約資訊寫入 advisory_bookings。諮詢內容可選擇性儲存至 meeting_transcripts (需用戶同意)。
5T 實踐點：
[T4 Transparent 透明]： 專家簡介、過往諮詢評價與收費標準清晰可見，確保資訊對稱。
[T2 Traceable 追溯]： 每次諮詢的預約紀錄、會議時間、參與者與費用都會被詳細記錄，方便企業管理諮詢預算與成效。

5. 功能項目解說和使用技術 (Features & Tech Stack)
專家篩選與搜尋： 支援多條件篩選（專長、語言、評分）。技術使用 Algolia 或 ElasticSearch 進行高效全文檢索與過濾。
線上預約系統： 處理時區轉換、衝突檢測與自動提醒。技術使用 FullCalendar.js 搭配後端排程服務（如 Google Calendar API 或自建排程邏輯）。
內建視訊會議： 提供安全、穩定的通話環境。技術使用 WebRTC 協議，搭配第三方服務（如 Daily.co, Twilio Video）或自建 SFU/MCU 服務。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線： 專家卡片在不同內容長度（如專長標籤數量）下，必須保持卡片高度一致，不可出現錯位或排版混亂。
🚨 邏輯/體驗紅線： 預約成功後，系統必須在 5 分鐘內發送確認郵件與行事曆邀請給雙方，若發送失敗必須有重試機制與管理員警報。

商情中心提供即時的 ESG 市場趨勢、競爭者分析、法規更新與產業報告，幫助企業掌握外部環境變化，及時調整永續策略，保持市場競爭力。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 永續長 (CSO) 或策略規劃人員。他們需要持續監控 ESG 領域的最新發展，但資訊來源分散、更新頻繁，難以高效篩選與分析。
體驗高光時刻 (Aha Moment)： CSO 在商情中心看到一篇關於「歐盟碳邊境調整機制 (CBAM) 最新實施細則」的深度分析報告，系統還自動標註了該細則對公司產品出口的潛在影響，讓 CSO 感到「決策所需資訊已自動送達」的價值。
操作軌跡：
1. 進入商情中心，瀏覽最新 ESG 新聞與報告。
2. 根據產業、地區或 ESG 議題篩選資訊。
3. 點擊感興趣的報告，閱讀摘要或下載完整文件。
4. 設定關鍵字追蹤，接收相關資訊的即時通知。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 採用「左側篩選與導航 + 中央新聞流 + 右側熱門議題趨勢圖」的三欄佈局。
核心液態玻璃元件： BrandNewsCard (包含標題、來源、發布時間、關鍵字標籤)、BrandTrendChart (顯示議題熱度趨勢)、BrandAlertSetting (用於設定通知)。
行動端適配 (RWD)： < 768px 時，左側篩選與右側趨勢圖收合為底部 Tab 或可滑動抽屜，中央新聞流保持單欄滾動，確保閱讀體驗。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 透過外部 API（如新聞聚合服務、法規資料庫）抓取數據，經由後端處理與分析後，儲存至 market_intelligence_feed。用戶設定寫入 user_preferences。
5T 實踐點：
[T5 Timely 即時]： 資訊來源每日自動更新，確保用戶獲取的是最新鮮的市場情報。
[T4 Transparent 透明]： 每條資訊都明確標示原始來源與發布時間，增加可信度。

5. 功能項目解說和使用技術 (Features & Tech Stack)
內容聚合與篩選： 從多個外部來源抓取、清洗與分類 ESG 相關資訊。技術使用 Python Scrapy 或 Puppeteer 進行網頁爬取，結合自然語言處理 (NLP) 進行內容分類與關鍵字提取。
趨勢分析與可視化： 識別熱門 ESG 議題並以圖表呈現其發展趨勢。技術使用 NLTK 或 SpaCy 進行文本分析，D3.js 或 ECharts 進行趨勢圖渲染。
個性化通知： 根據用戶設定的關鍵字或偏好，發送即時通知。技術使用 Web Push API 或 Email/SMS 服務（如 SendGrid, Twilio）。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線： 新聞卡片中的圖片載入失敗時，必須顯示預設佔位圖，不可出現圖片破圖或空白區域。
🚨 邏輯/體驗紅線： 關鍵字追蹤設定後，若在 24 小時內沒有任何相關新資訊，系統不應發送空通知，而應在通知中心提示「目前無符合您追蹤條件的新資訊」。

環境數據 Environmental Data
路徑： /environmental | 權限： PM, Data_Entry, ADMIN | 所屬旅程： II. 數據採集與分析

1. 模組定位 (Core Purpose)
環境數據模組是企業所有環境相關數據（如溫室氣體排放、能源消耗、水資源利用、廢棄物管理）的採集、管理、分析與可視化中心，為環境績效報告與減碳策略提供堅實數據基礎。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 各廠區數據填報員與永續專案經理 (PM)。填報員需要一個簡單的介面來輸入每月電費、水費等數據；PM 則需要確保數據的準確性與完整性，並能追蹤減碳進度。
體驗高光時刻 (Aha Moment)： 填報員上傳電費單後，系統自動識別關鍵數字並預填入表格，同時在右側顯示「本月碳排放預估值」，讓填報員感受到「數據輸入不再是枯燥的任務，而是有意義的貢獻」。
操作軌跡：
1. 選擇要填報的環境數據類別（例如：能源消耗）。
2. 選擇填報月份與廠區。
3. 輸入或上傳相關數據（如電費單 PDF）。
4. 系統自動計算並顯示初步結果。
5. PM 審核數據，確認無誤後提交。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 採用「左側數據類別導航 + 中央數據輸入表格/圖表 + 右側數據預覽與分析」的三欄佈局。
核心液態玻璃元件： BrandDataTable (支援行內編輯與驗證)、BrandFileUpload (帶有 OCR 預填功能)、BrandLineChart (顯示趨勢分析)。
行動端適配 (RWD)： < 768px 時，數據輸入表格自動轉化為「堆疊式表單 (Stacked Form)」，每個數據點獨立顯示；圖表則可透過點擊展開全螢幕視圖。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 數據從用戶輸入或 OCR 識別後，經由驗證寫入 environmental_data 資料表。原始文件儲存至 evidence_vault。計算結果影響 dashboard_kpis。
5T 實踐點：
[T1 Tangible 具體]： 將原始數據（如電費度數）自動轉換為具體的碳排放量，並以圖表形式呈現，讓數據影響更直觀。
[T2 Traceable 追溯]： 每一筆數據的輸入者、輸入時間、修改紀錄與審核狀態都清晰可查。
[T3 Transparent 透明]： 碳排放因子、計算公式等關鍵參數在系統中可供查閱，確保計算過程公開透明。

5. 功能項目解說和使用技術 (Features & Tech Stack)
OCR 數據預填： 自動從上傳的憑證中識別關鍵數字。技術使用 Google Cloud Vision API 或 Tesseract.js 進行文字識別，結合正則表達式進行數據提取。
數據驗證與異常檢測： 確保輸入數據的合理性與準確性。技術使用後端服務進行數據清洗與驗證，結合統計模型（如 Z-score）檢測異常值。
環境數據計算引擎： 根據不同排放源與活動數據，自動計算溫室氣體排放量。技術使用 Python 後端搭配 IPCC 排放因子資料庫。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線： 數據輸入表格的列數過多時，必須自動啟用水平滾動條，不可導致表格內容溢出或佈局錯亂。
🚨 邏輯/體驗紅線： 當用戶輸入的數據與歷史數據存在顯著偏差（例如：本月電費是上月 10 倍）時，系統必須彈出警告提示，要求用戶確認或提供解釋，防止誤輸入。

重大性矩陣 Materiality
路徑： /materiality | 權限： CSO, PM | 所屬旅程： III. 治理與確信

1. 模組定位 (Core Purpose)
用於執行雙重重大性評估 (Double Materiality)。將抽象的 ESG 議題，透過財務衝擊與環境衝擊分數，轉換為可視化、可排序的象限氣泡圖。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 永續長 (CSO) 或高階幕僚。過去必須收集數十份 Excel 問卷，手動計算平均值後再用 PowerPoint 辛苦拉泡泡圖，不僅耗時且容易算錯，開會時若被質疑分數來源，往往無法即時反查。
體驗高光時刻 (Aha Moment)： 在跨部門策略會議上，CSO 直接拉動網頁上的分數滑桿，下方的「溫室氣體排放」氣泡立刻平滑飛入右上角的「核心重大象限」。這種即時算繪的科技感，極大提升了 CSO 在會議中的專業權威感。
操作軌跡：
1. 匯入本年度候選的 ESG 議題清單。
2. 統整各方評分後，在介面上進行最終權重微調 (拉動 Sliders)。
3. 觀察氣泡動態落點，辨識出今年的 5 大核心議題。
4. 點擊「鎖定矩陣」，產出該年度不可篡改的策略基準。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 頂部為分數拉桿列 (Sliders Array)，下方 70% 區域為滿版的互動式二維氣泡圖。
核心液態玻璃元件： BrandSlider (具備數值即時反饋)、MatrixBubble (帶有 Liquid 模糊邊緣的氣泡)。
行動端適配 (RWD)： 透過 Media Query 將預設視圖切換為「議題優先級列表 (List View)」，並提供「點擊展開圖表」的橫向全螢幕選項，避免手指誤觸。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 分數變更即時運算 (X*Y 軸權重)，寫入 materiality_topics 表。
5T 實踐點：
[T1 Tangible 具體]： 將 1-5 分的抽象問卷評分，具體化為座標軸上的絕對落點與氣泡大小。
[T3 Trackable 追蹤]： 每次主管調整滑桿分數，系統自動在該議題的 Audit Log 中留下 Score_Changed 事件紀錄。

5. 功能項目解說和使用技術 (Features & Tech Stack)
互動式氣泡圖渲染： 需要支援流暢的座標移動動畫與 Tooltip 懸浮提示。技術使用 Recharts 搭配客製化 SVG 元素，或直接以 D3.js 實作力導向 (Force-directed) 防重疊邏輯。
雙重重大性狀態同步： 上方拉桿與下方圖表需保持毫秒級同步。技術使用 React useMemo 計算權重，結合 Zustand 或 Context API 進行狀態共享。
矩陣鎖定封印： 確認後禁止修改。技術使用 Supabase 的 Row Level Security (RLS) 變更資料表權限為 Read-Only。

1. 模組定位 (Core Purpose)
平台所有佐證文件（如電費單、ISO 證書、採購合約）的加密儲存與查核中心。這是支撐平台「可信度」的最底層防線。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 各部門填報窗口與外部稽核員。填報者常找不到最新版憑證，稽核員則總是懷疑「這張截圖有沒有被 Photoshop 修過？」。雙方處於不信任且溝通成本極高的狀態。
體驗高光時刻 (Aha Moment)： 填報者將 PDF 拖入區塊的瞬間，系統伴隨清脆音效亮起綠色的 ZKP Verified 與專屬 Hash 碼。稽核員看到這串由密碼學保證的標記時，立刻放下了戒心，雙方達成零摩擦的信任。
操作軌跡：
1. 填報窗口將供應商提供的 PDF 拖曳至 Dropzone。
2. 系統讀取檔案並展示 SHA-256 雜湊運算的動畫進度條 (增加科技儀式感)。
3. 選擇該文件要關聯的數據項目（例如：一月份台北廠電力）。
4. 檔案入庫，狀態轉為綠色的「已封印 (Sealed)」。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 採用「檔案總管型表格 (Data Table)」佈局，結合頂部區域的「全寬拖曳上傳區 (Dropzone)」。
核心液態玻璃元件： VaultOmniTable (萬能聖碑表格)、BrandStatusDot (顯示 ZKP 驗證狀態：綠=已封印，紅=檔案損毀/失效)。
行動端適配 (RWD)： 表格必須自動轉化為「堆疊式資料卡片 (Stacked Cards)」，隱藏複雜的 Hash 字串，僅保留檔名、GRI 條款與 5T 狀態燈號。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 實體檔案上傳至 Supabase Storage / S3，MetaData 與 SHA-256 Hash 寫入 evidence_vault 資料表。
5T 實踐點：
[T5 Trustworthy 信任]： 檔案拖入前端瀏覽器的瞬間，即刻計算出 SHA-256 Hash，並隨檔案上傳至資料庫。後續對圖片的任何微小修改，都會導致 Hash 對不上而亮起紅燈警報。

5. 功能項目解說和使用技術 (Features & Tech Stack)
客戶端 Hash 運算： 為了減少伺服器負擔與傳輸竄改風險，在前端直接算 Hash。技術使用瀏覽器原生的 Web Crypto API (crypto.subtle.digest)。
拖曳上傳互動： 支援多檔案與拖曳辨識。技術使用 react-dropzone 搭配客製化 Liquid Glass 樣式邊框高亮。
大檔案串流上傳： 避免超過 50MB 的 PDF 卡死瀏覽器。技術使用 Supabase Storage 的 TUS (Resumable Uploads) 協議進行分塊上傳。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線： 檔名或 Hash 字串過長時必須觸發 text-overflow: ellipsis (顯示...)，配合 Tooltip 檢視完整內容，絕對不可撐開表格導致整個頁面出現醜陋的 X 軸滾動條。
🚨 邏輯/體驗紅線： 若系統檢測到使用者上傳了重複的檔案 (Hash 完全相同)，必須阻斷上傳並彈出友善提示：「此文件已存在於金庫中，是否直接建立關聯？」，以節省儲存空間與防止資料冗餘。

4. 更新任務狀態（如從「待辦」拖曳到「完成」）。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)

核心液態玻璃元件： BrandKanbanBoard (支援拖曳排序)、BrandTaskCard (包含任務標題、截止日期、負責人、進度條)、BrandDatePicker (用於設定截止日期)。

[T1 Tangible 具體]： 將抽象的工作內容轉化為具體的任務卡片，並以看板形式視覺化呈現，讓工作進度一目瞭然。

5. 功能項目解說和使用技術 (Features & Tech Stack)
看板式任務管理： 支援任務卡片的拖曳、排序與狀態更新。技術使用 React Beautiful Dnd 或 DnD Kit 實現拖曳功能。
任務詳情與協作： 允許用戶查看任務描述、附件、評論，並與其他成員協作。技術使用 Supabase Realtime 實現實時評論與更新。
截止日期提醒： 在任務即將到期時自動發送提醒。技術使用後端排程服務（如 cron job）發送 Email 或系統通知。

🚨 UI 跑版紅線： 任務卡片中的標題或描述文字過長時，必須觸發 text-overflow: ellipsis，並提供 Tooltip 顯示完整內容，不可撐破卡片。
