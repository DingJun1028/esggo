VerifyLinkT / Audit Verification
路徑： /audit-verify | 權限： AUDITOR, COMPLIANCE_OFFICER, PM, ADMIN | 所屬旅程： V. 確信審計與發佈

1. 模組定位 (Core Purpose)
VerifyLinkT 是 ESG GO 的核心審計與確信模組，透過區塊鏈技術與零知識證明 (ZKP)，為永續報告的數據與內容提供不可篡改的驗證鏈，確保報告的真實性與合規性，提升企業的永續信任度。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 外部稽核員、內部法遵與永續長 (CSO)。他們面臨的痛點是傳統審計耗時費力，難以追溯數據源頭，且常因缺乏透明度而導致報告公信力不足，甚至引發「漂綠」質疑。
體驗高光時刻 (Aha Moment)： 當稽核員點擊報告中的某個數據點，系統即時彈出一個側邊抽屜，顯示該數據的完整「區塊鏈驗證路徑」，包含原始證據 Hash、上傳時間、修改紀錄與 ZKP 驗證狀態。稽核員看到這條清晰且不可竄改的信任鏈時，瞬間感受到「這份報告是真正可信的」巨大安心感。
操作軌跡：
1. 稽核員從儀表板進入「待審計報告」列表。
2. 選擇一份報告，進入 VerifyLinkT 介面。
3. 點擊報告中的「溫室氣體排放量」數據，查看其底層的 ZKP 驗證與證據金庫連結。
4. 檢閱系統自動生成的「合規性檢查報告」，確認無重大異常。
5. 點擊「簽署驗證」，完成該報告的數位確信流程。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 採用「左側報告導航樹 + 中央報告預覽區 + 右側驗證詳情抽屜」的三欄佈局。中央預覽區支援點擊互動，右側抽屜則顯示選定元素的驗證鏈與元數據。
核心液態玻璃元件： VerifyChainBadge (顯示 ZKP 狀態與 Hash 摘要)、AuditLogTimeline (時間軸形式的審計紀錄)、ReportViewer (高亮顯示可驗證數據點的報告預覽器)。
行動端適配 (RWD)： < 768px 時，左側導航樹與右側驗證詳情抽屜自動收合為「底部 Tab」或「懸浮 FAB」，中央報告預覽區必須 100% 滿版。報告預覽區的互動點擊會以全螢幕 Modal 顯示驗證詳情，確保資訊清晰可讀。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 從 `report_drafts` 讀取報告內容，從 `evidence_vault` 讀取佐證文件 Hash，從 `audit_logs` 讀取操作紀錄。驗證結果與簽署狀態寫入 `report_verification_status` 表，並將關鍵驗證 Hash 上鏈至私有區塊鏈 (或 Supabase Edge Functions 模擬的信任鏈)。
5T 實踐點：
[T1 Tangible 具體]： 將抽象的「信任」具體化為可視的「驗證鏈」與「ZKP 狀態徽章」，讓稽核員能直觀理解數據的可靠性。
[T2 Traceable 追溯]： 每個報告數據點都與其在 `evidence_vault` 中的原始證據 Hash 雙向連結，並記錄所有修改與驗證操作的完整時間戳與使用者 ID。
[T3 Trackable 追蹤]： 每次驗證操作、狀態變更或稽核員簽署，都會被記錄在不可篡改的 `audit_logs` 資料表中，形成完整的審計軌跡。
[T4 Transparent 透明]： 系統清楚揭示每個數據點的驗證邏輯、所依據的證據來源，以及 ZKP 的驗證過程，讓稽核員能完全理解其背後的信任機制。
[T5 Trustworthy 信任]： 結合區塊鏈技術（或基於 Supabase 的加密簽名與時間戳服務）與零知識證明 (ZKP)，確保數據一旦被驗證並「上鏈」，其完整性與真實性便無法被惡意竄改，提供最高等級的數據信任。

5. 功能項目解說和使用技術 (Features & Tech Stack)
報告內容互動式驗證： 允許稽核員點擊報告中的任何數據或聲明，即時觸發後端驗證流程並顯示結果。技術使用 React Query 進行數據預取與緩存，結合 WebSockets 實時更新驗證進度。
零知識證明 (ZKP) 整合： 對於敏感數據（如薪資總額），允許在不揭露原始數值的情況下，證明其符合特定條件（如「總額大於某閾值」）。技術使用 SnarkJS 或 Circom 庫在後端生成與驗證 ZKP 證明，前端僅顯示證明結果。
區塊鏈信任鏈： 將關鍵數據的 Hash 與驗證狀態「上鏈」，提供不可篡改的信任錨點。技術使用 Hyperledger Fabric 或基於 Supabase Functions 與 PostgreSQL 的自定義信任鏈實現，確保數據的不可否認性。
自動合規性檢查： 根據預設的 GRI/ISSB 準則，自動掃描報告內容，標記潛在的合規風險或遺漏點。技術使用自然語言處理 (NLP) 服務（如 Google Cloud Natural Language API）進行文本分析與關鍵詞匹配。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線： 當報告內容過長或包含複雜表格時，中央報告預覽區的滾動條必須正常運作，且右側驗證詳情抽屜的內容不可溢出或被遮擋。ZKP 狀態徽章必須在任何螢幕尺寸下保持清晰可見。
🚨 邏輯/體驗紅線：
1. 若 ZKP 驗證失敗或證據 Hash 不匹配，系統必須以醒目的紅色警示，並提供詳細的失敗原因，而非靜默失敗或顯示模糊錯誤。
2. 稽核員在點擊「簽署驗證」後，必須彈出包含所有驗證結果摘要的二次確認對話框，並在簽署完成後，該報告的驗證狀態必須立即更新為「已確信」，且不可再被未經授權的用戶修改。
3. 斷網情況下，必須提供「離線審閱模式」，允許稽核員瀏覽已緩存的報告內容和驗證紀錄，並在網路恢復後自動同步簽署意圖，避免資料遺失或操作中斷。
4. 任何嘗試繞過 VerifyLinkT 直接修改已確信報告內容的行為，必須被系統嚴格阻斷，並在 `audit_logs` 中記錄為「安全事件」。
