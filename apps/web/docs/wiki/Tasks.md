# 任務中心 Tasks [Tasks]
路徑： /tasks | 權限： PM, ALL_USERS | 所屬旅程： III. 執行與協作

1. 模組定位 (Core Purpose)
任務中心是 ESG GO 的核心協作中樞，提供跨部門任務分派、進度追蹤與 5T 合規檢查，確保 ESG 策略從規劃到執行的無縫落地。

2. 客戶旅程與 UX 體驗 (Customer Journey & UX)
目標對象與痛點： 永續專案經理 (PM) 或永續辦公室成員。他們經常需要協調多個部門（如人資、採購、廠務）來收集數據或執行專案，但傳統的郵件或試算表溝通效率低下，任務容易被遺忘或延誤，導致報告撰寫時才發現數據缺失。
體驗高光時刻 (Aha Moment)： 當 PM 在任務看板上看到所有部門的任務進度一目了然，並且系統自動標示出哪些任務的數據已通過 5T 檢查，讓 PM 感到一切盡在掌握，不再需要頻繁催促。
操作軌跡：
1. PM 進入任務中心，快速瀏覽各部門的任務看板，發現「溫室氣體盤查」任務進度落後。
2. 點擊進入該任務詳情頁，看到系統自動標註「採購部：供應商碳足跡數據」任務尚未完成，且 5T 檢查未通過。
3. PM 直接在任務下方留言區 `@採購部負責人`，並附上相關指引文件。
4. 採購部負責人收到通知，上傳數據並標記完成，PM 看到任務狀態即時更新為「待審核」，並在 5T 檢查通過後自動轉為「已完成」。

3. UI/UX 視覺與 RWD 基準 (Visual & RWD Baseline)
桌面版佈局： 採用「看板式佈局 (Kanban Board)」，主區域為可自定義的任務列（如「待辦」、「進行中」、「待審核」、「已完成」），每列寬度彈性但總和不超過父容器，可垂直滾動。任務卡片支援拖曳。頂部固定為篩選與新增任務區。
核心液態玻璃元件： TaskCard (顯示任務名稱、負責人頭像、截止日、進度條、5T 狀態燈號)、BrandProgressRing (任務進度環)、BrandStatusBadge (5T 檢查結果)、BrandButton (用於新增/編輯任務、篩選)、BrandModal (用於任務詳情與編輯)、BrandAvatar (負責人頭像)、BrandDropdown (用於任務篩選或更多操作)。
行動端適配 (RWD)： < 768px 時，看板列自動堆疊為垂直滾動列表，並隱藏列標題，僅顯示任務卡片。任務卡片內容精簡，僅顯示核心資訊（任務名稱、負責人、截止日、進度條、5T 狀態燈號），點擊卡片展開詳細資訊。拖曳功能在行動端透過長按觸發，或在特定視圖下禁用以簡化操作。

4. 核心邏輯與 5T 協定 (Logic & 5T Protocol)
資料流向： 任務的建立、更新、狀態變更等操作，將寫入 `tasks` 資料表。任務與 `users` 表關聯負責人，與 `projects` 表關聯專案。任務完成後，若涉及數據上傳，會觸發對應數據點的 5T 檢查，並將結果儲存於 `task_compliance_status` 表，同時更新 `tasks` 表中的 5T 狀態欄位。任務的留言互動則寫入 `task_comments` 表。這些數據將匯總至 ESG Dashboard 的「任務進度」KPI。
5T 實踐點：
[T1 Tangible 具體]： 每個任務都必須關聯到一個具體的 ESG 指標、數據點或報告章節，避免任務目標模糊不清。
[T2 Traceable 追溯]： 任務的建立、指派、狀態變更、留言互動，都應有完整的操作日誌 (Audit Log)，記錄操作者、時間與變更內容。
[T3 Trackable 追蹤]： 任務的建立、指派、狀態變更、留言互動，都應有完整的操作日誌 (Audit Log)，記錄操作者、時間與變更內容，並可透過任務詳情頁的「活動紀錄」追溯所有歷史變動，確保過程可審計。
[T4 Transparent 透明]： 任務的進度、負責人、截止日以及 5T 檢查結果，都應在看板上清晰可見，讓所有相關人員都能掌握最新狀態。
[T5 Trustworthy 信任]： 任務完成後，若涉及數據上傳，系統會自動觸發對應數據的 5T 檢查（例如：檢查數據來源的 Hash 值是否與證據金庫一致），確保數據的可靠性。

5. 功能項目解說和使用技術 (Features & Tech Stack)
任務看板與拖曳排序： 支援任務卡片在不同看板列之間拖曳移動，以及在同一列內調整優先級。技術使用 `react-beautiful-dnd` 或 `dnd-kit` 搭配 Zustand 進行前端狀態管理。
任務建立與編輯表單 (Task Creation & Editing Form)： 支援任務名稱、描述、負責人、截止日、關聯 ESG 指標、附件上傳等欄位。技術使用 `React Hook Form` 進行表單管理與驗證，搭配 `react-datepicker` 選擇日期，`react-select` 選擇負責人與關聯指標。
5T 合規自動檢查： 當任務狀態變更為「待審核」或「已完成」時，後端自動觸發與該任務關聯數據的 5T 檢查邏輯。技術使用 Node.js (NestJS) 或 Python (FastAPI) 搭配背景 Job Queue (e.g., BullMQ, Celery) 執行非同步檢查，結果透過 WebSocket 或 Supabase Realtime 推送至前端。
跨模組數據關聯： 任務可直接關聯到「證據金庫」中的文件或「數據中心」的數據點，方便追溯與查核。技術使用 Supabase Foreign Key 關聯，前端使用 `react-query` 進行數據預取和緩存。
即時通知與協作： 任務指派、狀態變更、留言 `@` 負責人時，系統發送即時通知（站內信、Email）。技術使用 Supabase Realtime 或 Pusher 實現即時通知，搭配 `react-toastify` 顯示前端提示。

6. 品質達標與驗收紅線 (QA Red Lines)
🚨 UI 跑版紅線： 當單一任務卡片內容（如任務描述）過長時，必須自動截斷並顯示「...」或提供展開按鈕，不可撐破卡片容器或導致看板列寬度異常。任務拖曳時，拖曳陰影與目標位置的視覺回饋必須流暢且無閃爍。在不同解析度下，看板列的數量與寬度必須保持合理，不可出現單列過窄或過寬導致內容難以閱讀的情況。空看板列應顯示友善的「無任務」提示，而非空白。
🚨 邏輯/體驗紅線： 若 5T 合規檢查結果為「不通過」，系統必須清晰標示出具體不通過的原因（例如：T5 Hash 不符、T1 數據點缺失），而非僅顯示「失敗」。當使用者嘗試指派任務給已停用或不存在的帳號時，必須即時給予錯誤提示，而非允許指派後才發現通知無法送達。多個使用者同時編輯同一任務時，系統應提供衝突解決機制或即時同步更新，避免資料覆蓋。斷網情況下，任務狀態變更應暫存於本地 (IndexedDB)，待連線恢復後自動同步，並給予使用者明確的離線工作提示。
